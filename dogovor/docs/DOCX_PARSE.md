# Парсинг шаблона договора .docx

**Зачем:** безопасно перенести изменения из обновлённого `ПАЙ_ОПТИКС_МЕД_ДОГ.docx` в HTML-шаблон `backend/templates/contract.html` без потери данных. Исходный .docx не изменяется — скрипт только читает файл.

## Автоматизация (одна команда)

Положите обновлённый договор **в папку проекта** `dogovor/contract_template/` с именем `ПАЙ_ОПТИКС_МЕД_ДОГ.docx` — так он не потеряется при переносе проекта. Затем:

```bash
cd dogovor/scripts
python3 sync_from_docx.py
```

Скрипт по умолчанию читает `dogovor/contract_template/ПАЙ_ОПТИКС_МЕД_ДОГ.docx`. Или укажите свой путь:

```bash
python3 sync_from_docx.py "/путь/к/договор.docx"
```

Скрипт создаст/обновит `backend/templates/contract_extracted.html`. Дальше откройте его и `contract.html` и перенесите нужные правки (новые формулировки, пункты), не трогая переменные `{{ ... }}` и блоки `{% if %}`.

## Как обновить шаблон после правок в .docx (пошагово)

1. Запустите `sync_from_docx.py` (см. выше).
2. Откройте `contract_extracted.html` и `contract.html`, сравните (diff или визуально).
3. Перенесите в `contract.html` новые/изменённые абзацы, сохраняя переменные Jinja2.
4. Файл `contract_extracted.html` в .gitignore — не коммитить.

## Переменные в шаблоне

- `{{ contract_number }}`, `{{ current_date }}`, `{{ city }}`, `{{ cabinet_address }}`
- `{{ patient_fio }}`, `{{ patient_birth_date }}`, паспорт, `{{ reg_address }}`
- `{{ staff_fio }}`, `{{ staff_signature_url }}`, `{{ staff_signature_data_url }}` — подпись оптометриста (data URL с ресайзом для PDF)
- `{{ patient_signature_data_url }}` — подпись клиента (в 4 местах: основной блок, приложения 1–3)

Новые поля из .docx добавляйте в `context` в `app/api/contracts.py` и в шаблон.

## Нумерация (раздел 1, подраздел 1.1, 1.2…)

В Word в оригинале используются **нумерация** (1., 1.1., 1.2., 2.1. …), а не маркеры. Текущий скрипт `docx_to_html` и библиотека **python-docx не отдают номера пунктов списка** — в XML они лежат в `word/numbering.xml` и в `w:pPr/w:numPr` параграфа, готового API нет.

**Варианты:**

1. **Ручная нумерация в шаблоне (сделано)**  
   В `contract.html` пункты оформлены явным текстом: `1.1.`, `1.2.`, `2.1.` и т.д. При изменении текста договора номера правятся вручную. Надёжно и не зависит от формата .docx.

2. **Парсинг XML .docx вручную**  
   Читать из каждого параграфа `w:numId`, `w:ilvl`, по ним в `word/numbering.xml` брать `lvlText`/`numFmt` и считать номер. Реализация сложная, хрупкая при смене стилей в Word.

3. **docx2python**  
   Вызов `docx2python(path).body` иногда выводит нумерацию в текст, но не для всех форматов и уровней вложенности.

4. **Mammoth (docx → HTML)**  
   Конвертирует .docx в HTML; нумерация может попасть в разметку или в текст. Для нашего случая неудобно: теряется контроль над переменными Jinja2, нужно снова вставлять `{{ ... }}`.

5. **Макрос в Word: ConvertNumbersToText**  
   В Word выполнить макрос, превращающий нумерацию в обычный текст, сохранить копию .docx и затем использовать `sync_from_docx` — в извлечённом HTML появятся числа в тексте. Удобно для одноразового переноса структуры.

**Рекомендация:** держать нумерацию в `contract.html` вручную (вариант 1). При крупных правках по .docx можно временно применить вариант 5 и сверять номера с шаблоном.
