# Рекомендуемый вариант: распознавание паспорта с учётом безопасности

## Текущее состояние проекта

- **Фото паспорта:** не сохраняем (по README — удаление с фронта в течение ~30 сек после OCR).
- **В БД (dogovor_contracts):** только `contract_number`, `location_id`, `staff_id`, **patient_fio**, `pdf_path`, `device_uuid`. Серия, номер, даты, адрес и т.д. **в таблицу не пишутся** — только в договор (PDF).
- **Хранение паспортных данных:** по сути только внутри сгенерированного PDF (договор) в Storage. Это соответствует требованию «паспортные данные не хранить нигде кроме нашего договора».

Итог: **сам паспорт (фото) нигде не храним; текстовые данные из паспорта попадают только в договор (PDF).** Эту модель нужно сохранить при подключении стороннего сервиса.

---

## Требование по безопасности

- **Паспортные данные в идеале не хранить нигде кроме нашего договора** — подтягивать их для заполнения, но **сам паспорт (изображение) не хранить**.
- При использовании стороннего API: минимизировать хранение фото у провайдера; у себя не сохранять снимок паспорта ни в БД, ни в Storage, ни на диск.

---

## Лучший вариант для проекта

### Схема: сторонний API через наш бэкенд, фото только в памяти

**Идея:** стабильное распознавание через специализированный сервис (паспорта РФ), при этом **фото паспорта не сохраняется у нас** и по возможности не хранится у провайдера дольше обработки.

1. **Сервис:** **Биорг.Паспорт** (beorg.ru) — паспорта РФ/СНГ, прописка (в т.ч. рукописная), обработка в РФ, ФЗ-152, есть режим без длительного хранения / on‑premise при необходимости.
2. **Поток данных:**
   - Пользователь снимает разворот (и при необходимости прописку) на фронте.
   - Фронт отправляет фото **на наш бэкенд** (например, `POST /api/v1/passport/recognize`, multipart).
   - Бэкенд **не пишет фото на диск и не кладёт в БД/Storage.** Принимает файл в память, сразу вызывает API Биорг (или другой выбранный сервис), получает JSON с полями (ФИО, дата рождения, серия, номер, кем выдан, дата выдачи, адрес и т.д.), отдаёт этот JSON на фронт и забывает фото.
   - Фронт подставляет ответ в форму; пользователь при необходимости правит и переходит к подписи и генерации договора.
   - При генерации договора те же данные (уже из формы) попадают только в PDF и в письмо; в БД по-прежнему пишем только то, что сейчас (в т.ч. только `patient_fio`, без серии/номера в таблице).

3. **Прописка:** Биорг обрабатывает **оба разворота в одном запросе**. Тип документа `PASSPORT_REG`: в `images` передаются два base64 — разворот 2–3 стр. и страница с пропиской; в ответе приходит поле `Address` (адрес регистрации). Отдельно «придумывать» второй разворот не нужно — сервис сам распознаёт прописку по второму фото.

4. **Политика у провайдера:** при подключении Биорг (или Regula) уточнить в договоре/документации: **не сохраняют ли они изображения паспортов после обработки.** Если есть опция «обработка без сохранения» или «удаление после ответа» — включить. Вариант **on‑premise** (Биорг) даёт полный контроль: данные не покидают ваш контур.

5. **Валидация серии/номера:** после того как серия и номер получены (из API или ручным вводом), вызывать **DaData** (проверка формата и база недействительных паспортов МВД). Это не хранение паспорта, а проверка уже введённых реквизитов — допустимо с точки зрения «паспорт не хранить».

---

## Почему именно так

| Критерий | Решение |
|----------|---------|
| **Стабильные серия/номер** | Специализированный API (Биорг), а не Tesseract, у которого вертикальный текст в паспорте РФ не читается. |
| **Паспорт (фото) не хранить** | Фото идёт только в память бэкенда → один запрос к API → ответ → фото не сохраняем ни в файл, ни в БД, ни в Storage. В БД оставляем только текущий минимум (в т.ч. только ФИО из паспортных данных). |
| **Данные только в договоре** | Текстовые поля из паспорта используются только для подстановки в форму и в итоговый PDF (договор). В таблицу договоров не добавляем серию/номер/даты — только то, что уже есть. |
| **Безопасность и ФЗ-152** | Провайдер в РФ (Биорг), при необходимости on‑premise или режим без хранения изображений. Мы со своей стороны не храним снимок паспорта. |
| **Один поток для пользователя** | Два шага съёмки (разворот + прописка) сохраняются; вместо Tesseract для разворота — один запрос к нашему API, который внутри дергает Биорг. Фронт не усложняется. |

## Альтернативы (если появятся ограничения)

- **Regula Document Reader** — если нужна поддержка многих типов документов или VizOCR по РФ; схема та же: наш бэкенд, фото только в памяти, ответ — в форму, без сохранения снимка.
- **On‑premise Биорг** — если нельзя отправлять фото в облако: развернуть у себя, тогда паспорт не покидает вашу инфраструктуру.
- **Резерв:** оставить Tesseract как «попробовать бесплатно» и кнопку «Распознать через сервис» (отправка на наш API → Биорг), если серия/номер не распознались или пользователь хочет точнее.

## Что сделать в коде (кратко)

1. **Бэкенд:** эндпоинт `POST /api/v1/passport/recognize` (multipart: файл разворота, опционально второй файл — прописка). В обработчике: принять файл в память (не сохранять на диск), вызвать API Биорг (ключ из env), смапить ответ к полям `PatientData`, вернуть JSON. Логировать только факт вызова без ПДн/фото.
2. **Фронт:** после выбора фото разворота (и при необходимости прописки) вызывать этот эндпоинт вместо Tesseract (или в дополнение, с кнопкой «Через сервис»). Подставлять ответ в те же поля формы, что и сейчас.
3. **Конфиг:** переменные окружения типа `PASSPORT_API_URL`, `PASSPORT_API_KEY` (или аналог под выбранный сервис). При отсутствии ключа эндпоинт может возвращать 503 или не регистрироваться — тогда фронт использует только Tesseract/ручной ввод.
4. **Опционально:** после заполнения серии/номера (из API или вручную) вызов DaData для проверки паспорта и отображение «Паспорт проверен» / предупреждения.

Данные из паспорта по-прежнему не хранятся нигде, кроме подстановки в договор (PDF); сам паспорт (изображение) ни у нас, ни по возможности у провайдера не сохраняется.
