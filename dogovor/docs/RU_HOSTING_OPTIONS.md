# Российские аналоги Railway (на будущее)

Если понадобится переезд с Railway в российское облако (по юридическим/инфраструктурным причинам), ниже — варианты и что потребуется.

## Что нам нужно от платформы

- HTTPS + домен
- Переменные окружения (Supabase/SMTP/CORS и т.д.)
- Удобный деплой из Git или контейнеров
- (Опционально) Managed Postgres / Object Storage

## Варианты

### Yandex Cloud

- **Похоже на Railway:** меньше; это скорее облако с сервисами.
- **Как запускать:** чаще всего через контейнер:
  - Serverless Containers (просто выкатывать контейнер)
  - или Compute (виртуалка)
  - или Managed Kubernetes (сложнее)
- **Плюсы:** дата-центры в РФ, много управляемых сервисов (Postgres, Object Storage), хорошая интеграция.
- **Минусы:** выше порог входа, чем Railway (IAM, сеть, контейнеризация).

### VK Cloud

- **Похоже на Railway:** ближе, чем классические IaaS (есть App Platform/контейнерные сценарии).
- **Плюсы:** РФ, варианты PaaS/контейнеры.
- **Минусы:** всё равно чаще потребуется Docker/настройки деплоя.

### Selectel / SberCloud / Timeweb Cloud

- **Как запускать:** чаще через Docker/виртуалки/Kubernetes.
- **Плюсы:** РФ, разные тарифы и инфраструктура.
- **Минусы:** обычно менее «one-click», чем Railway.

## Что нужно подготовить для переезда (когда созреешь)

1. **Контейнеризация**: Dockerfile для бэкенда и фронта (или фронт — статика + CDN).
2. **Переменные окружения**: перенести env в секреты новой платформы.
3. **Домен и TLS**: настроить DNS и сертификаты (обычно автоматом/через LB).
4. **Сеть**: CORS/allowed origins под новый домен.
5. **Данные**: Supabase можно оставить как есть (если устраивает география), либо мигрировать БД/Storage в российские аналоги.

Сейчас рационально: довести продукт и UX на Railway, а переезд делать отдельной задачей, когда появятся чёткие требования (география данных, SLA, бюджет).
