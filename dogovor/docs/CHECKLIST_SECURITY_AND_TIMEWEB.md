# Чеклист: безопасность и переезд на Timeweb + Яндекс

Перед переездом на Timeweb: проверить безопасность, ничего не забыть из плана, затем миграция и настройка почты.

---

## Результаты проверки по коду (пройдено)

- **.env:** в корне репо `.gitignore` есть `.env` и `.env.local` — файлы не попадут в git. На проде секреты только в переменных хостинга.
- **Supabase:** `SUPABASE_SERVICE_KEY` используется только в бэкенде (`app/db.py`, `config.py`). Во фронте только `NEXT_PUBLIC_DOGOVOR_API_URL` — ключей Supabase нет.
- **Beorg:** переменные `BEORG_*` читаются только в бэкенде (`config.py`, `beorg_recognize.py`, `passport.py`).
- **Почта:** пароли/ключи только из env (`config.py`: `smtp_password`, `resend_api_key`); в коде нигде не захардкожены.
- **CORS:** бэкенд берёт `CORS_ORIGINS` из env и парсит список (`app/config.py`, `main.py`). На проде нужно задать URL фронта.
- **Приложение 2:** в шаблоне договора `contract.html` есть блок «Согласие пациента на обработку персональных данных» (152-ФЗ). Юридическую полноту при необходимости уточнить.

## 1. Проверка безопасности (по `SECURITY_DEPLOY.md`)

### Секреты и переменные окружения
- [x] `.env` не в git (уже в `.gitignore`). На проде все секреты только в переменных хостинга.
- [x] **Supabase:** `SUPABASE_SERVICE_KEY` (service_role) только на бэкенде, не светить на фронт.
- [x] **Beorg:** `BEORG_PROJECT_ID`, `BEORG_TOKEN`, `BEORG_MACHINE_UID` только на бэкенде.
- [x] **Почта:** пароль SMTP/Resend только в env, не в коде. Если ключ попал в логи/скрин — перевыпустить и заменить.
- [ ] **Supabase для прода:** по возможности использовать отдельный проект (или отдельную БД) от тестов. На Timeweb задать его `SUPABASE_URL` и `SUPABASE_SERVICE_KEY`. Если один проект на всё — просто не смешивать тестовые данные с боевыми.

### Сеть и доступ
- [ ] Бэкенд и фронт в проде доступны по **HTTPS**. На Timeweb при привязке домена к сервису HTTPS (Let's Encrypt) обычно настраивается автоматически. Проверить после деплоя.
- [ ] **CORS:** в переменных бэкенда задана `CORS_ORIGINS` — URL фронта через запятую (например `https://dogovor.pyeoptics.com`). Без этого браузер заблокирует запросы с фронта к API.
- [ ] При необходимости: ограничение доступа к API (IP, VPN или простая авторизация для планшетов).

### Хостинг и данные
- [ ] Переменные окружения задаются через настройки хостинга (Railway/Timeweb), не через `.env` в репо.
- [x] Политика конфиденциальности и согласие на обработку персональных данных учтены (в интерфейсе уже есть приложение 2 к договору).
- [ ] **Резервное копирование БД:** Supabase Dashboard → Project Settings → Database (Backups) или Table Editor → Export. Подробнее: `docs/SECURITY_DEPLOY.md`.

## 3. Переезд на Timeweb и настройка Яндекс-почты

### 3.1 Подготовка (до переезда)
- [ ] Решить, где будет жить бэкенд на Timeweb (VPS, облако, панель — в зависимости от тарифа). **Подробный пошаговый план:** `docs/TIMEWEB_MIGRATION_PLAN.md`.
- [ ] Сохранить текущие переменные окружения с Railway (Supabase, Beorg, CORS и т.д.) — понадобятся на Timeweb.
- [ ] Убедиться, что на Timeweb исходящий SMTP (порты 465/587) не блокируется.

### 3.2 Деплой бэкенда на Timeweb
- [ ] Развернуть бэкенд (Docker или `pip install` + uvicorn). Root Directory / рабочая директория: `dogovor/backend`. Пошагово: `docs/DEPLOY_TIMEWEB.md`.
- [ ] Задать все переменные окружения как на Railway:  
  `SUPABASE_URL`, `SUPABASE_SERVICE_KEY`, `STORAGE_BUCKET` (=`contracts`),  
  `BEORG_PROJECT_ID`, `BEORG_TOKEN`, `BEORG_MACHINE_UID`,  
  `CORS_ORIGINS` (URL фронта после переезда),  
  опционально: `PREPROCESS_PASSPORT_IMAGE`.
- [ ] **Почта — переключить на Яндекс:**  
  - `EMAIL_PROVIDER=smtp`  
  - `SMTP_HOST=smtp.yandex.ru`  
  - `SMTP_PORT=465`  
  - `SMTP_USER` — полный email (например `support@pyeoptics.com`)  
  - `SMTP_PASSWORD` — пароль приложения Яндекса (не основной пароль).  
  - `SMTP_FROM_EMAIL` — тот же ящик (например `support@pyeoptics.com`)  
  - `SMTP_FROM_NAME=Пай Оптикс` (или как нужно)
- [ ] Проверка почты после деплоя: открыть `https://ТВОЙ-БЭКЕНД-НА-TIMEWEB/debug/smtp-check`. Должны быть `smtp_configured: true`, `connection_ok: true`. См. `docs/EMAIL_SMTP.md`.

### 3.3 Фронт после переезда
- [ ] В настройках фронта (Vercel/Railway/Timeweb — где он будет) задать `NEXT_PUBLIC_DOGOVOR_API_URL` = новый URL бэкенда на Timeweb (без слэша в конце).
- [ ] В бэкенде в `CORS_ORIGINS` указать URL фронта (через запятую, если несколько).

### 3.4 Итог по почте
- **На Railway:** использовали Resend (`EMAIL_PROVIDER=resend`).
- **На Timeweb:** ставим `EMAIL_PROVIDER=smtp` и SMTP_* (Яндекс). Код менять не нужно — переключение только переменными окружения.

## 4. Быстрая проверка после переезда

1. Открыть фронт → выбрать кабинет и врача → сканирование (или ввод вручную) → проверка → подпись → «Подписать и отправить» с указанием своего email.
2. Убедиться, что письмо с PDF пришло и открывается.
3. При необходимости проверить `CORS_ORIGINS` и `NEXT_PUBLIC_DOGOVOR_API_URL`, если запросы к API падают.
