# Оценка: хватает ли информации для самостоятельной реализации «как у сервиса»

## Что уже есть в проекте и в документации

### 1. Структура МЧЗ (внутренний паспорт РФ)
- **Есть:** описание второй строки 44 символа — позиции 1–9 (серия+номер), 10 (контрольная), 11–13 (RUS), 14–19 (дата рождения), 20, 21 (пол), 22–28, **29 (4-я цифра серии)**, 30–41 (дата выдачи + код подразделения), 42–44. Источник: Приказ МВД 851, КонсультантПлюс.
- **Есть:** парсинг серии/номера из второй строки в коде (`parseSeriesNumberFromMRZ`, `parseMRZRawForSeriesNumber`).
- **Частично:** алгоритм контрольной цифры упомянут («Модуль 10», вес 731), но **нет пошаговой формулы** в наших доках (как именно умножать и суммировать). Формулу можно взять из КонсультантПлюс / ГОСТ 7501-1 или из открытых парсеров МЧЗ.
- **Нет в доках:** явное описание **первой строки** МЧЗ (тип PN, RUS, фамилия<<имя в модернизированном клере). Для полного парсера МЧЗ (ФИО из первой строки) это нужно искать в том же приказе.

### 2. Расположение зон на развороте (качественно)
- **Есть:** «МЧЗ — нижняя четверть 3-й страницы», «верхняя строка МЧЗ 17,9 мм от нижнего края», «длина полосы до 114 мм». Это привязка к **документу в мм**, не к пикселям фото.
- **Есть:** понимание, что серия/номер на развороте — вертикальный столбик, визуально в левой части (страница с фото).
- **Нет:** точной **разметки в мм** для всех полей разворота (где именно по X,Y заканчивается фото, начинается блок ФИО, где точно серия/номер, границы МЧЗ по вертикали). Официальные размеры есть в Постановлении №687 и приложениях к приказам МВД, но мы их **не выписали** в проект.

### 3. Алгоритмы и пайплайн
- **Есть:** общее описание того, как устроены сервисы (детекция документа → перспектива → регионы → OCR по зонам). Есть ссылки на статьи (document detection, perspective correction, MRZ detection).
- **Есть:** ссылки на open-source (paSport_eye, paspread, Russian_passport_text_recognition, RussianDocsOCR).
- **Нет:** пошаговой **спеки пайплайна** под наш стек (браузер или бэкенд): какой именно алгоритм детекции (контур по градиенту, Hough, готовые библиотеки), как задать регионы **относительно найденного документа** (не от кадра), какой PSM/настройки Tesseract для каждой зоны. То есть «рецепта» под свою реализацию нет.

### 4. Код и стек
- **Есть:** Tesseract.js на фронте, предобработка (ресайз, контраст), вырезы по фиксированным процентам, парсеры (разворот, МЧЗ, прописка). Это база, но не полноценный пайплайн с детекцией и привязкой к документу.
- **Нет в репо:** детекция контура документа (OpenCV.js или аналог), гомография/выравнивание перспективы, расчёт регионов от углов документа. В open-source проектах это обычно OpenCV (Python); для браузера пришлось бы портировать или брать готовые JS-библиотеки (например, для document scan).

### 5. Тестовые данные и валидация
- **Нет:** набора размеченных фото разворота/прописки (с известными полями) для автоматических тестов и подбора порогов. Без этого сложно объективно сравнивать варианты вырезов и парсеров.

## Чего не хватает для реализации «аналога сервиса»

| Блок | Что нужно | Где взять / сложность |
|------|-----------|------------------------|
| **Детекция документа** | Алгоритм или библиотека: найти четыре угла паспорта в кадре. | OpenCV (findContours, approxPolyDP, или готовые туториалы document scanner). В браузере — OpenCV.js или порт. Реализуемо по публичным материалам. |
| **Перспектива** | По 4 точкам — матрица гомографии, выровнять изображение в прямоугольник. | Классика, есть в OpenCV (getPerspectiveTransform, warpPerspective). В JS — библиотеки или свой расчёт. Реализуемо. |
| **Шаблон полей в координатах документа** | После выравнивания документ имеет фиксированные пропорции (разворот 2–3 стр. паспорта РФ). Нужны **относительные координаты** (0–1 или мм при известном DPI): где левая/правая страница, где МЧЗ, где блок ФИО, где серия/номер. | Официальные описания (Постановление 687, приложения МВД) + один эталонный разворот с замером. Мы **не выписали** эти координаты. Нужна ручная разметка по эталону или из нормативов. Средняя сложность. |
| **Сегментация/маски по зонам** | По шаблону вырезать регионы из выровненного изображения и отдавать в OCR (для МЧЗ — одна/две строки, для серии/номера — узкая полоса с поворотом и т.д.). | Реализуемо, если есть шаблон (см. выше). |
| **Настройки OCR по зонам** | PSM и при необходимости язык под каждую зону (например, только цифры, одна строка для МЧЗ). | Документация Tesseract; у нас не зафиксировано. Низкая сложность. |
| **Контрольная сумма МЧЗ** | Точная формула 731 для позиций 1–9 и при необходимости для других полей — для отсечения мусора и проверки. | ГОСТ / КонсультантПлюс или открытые парсеры МЧЗ. Реализуемо. |
| **Парсинг первой строки МЧЗ** | ФИО из модернизированного клера (транслит) — опционально, для дублирования/проверки полей с визуальной зоны. | Спека в приказе МВД; таблица транслитерации. Средняя сложность. |
| **Прописка** | Структура страницы с пропиской (где блок адреса), возможная рукописная часть. | У нас только эвристики по меткам; точной разметки страницы прописки нет. Для «как у сервиса» нужна либо разметка, либо отдельная модель/логика. |

## Итоговая оценка

**Вопрос:** есть ли вся информация для самостоятельной реализации такого сервиса?

**Ответ: нет, не вся.**

- **Что есть и достаточно:**  
  - структура второй строки МЧЗ и парсинг серии/номера (и при желании дат, пола из МЧЗ);  
  - качественное описание расположения зон (МЧЗ внизу, серия/номер слева и т.д.);  
  - общее понимание пайплайна (детекция → перспектива → регионы → OCR);  
  - ссылки на алгоритмы (OpenCV, статьи) и open-source проекты.

- **Чего не хватает:**  
  - **жёсткой привязки полей к документу:** точных координат/шаблона разворота 2–3 стр. (в мм или относительных) из нормативов или эталона;  
  - **пошаговой спецификации пайплайна** под наш стек (какой детектор, как из углов документа получать регионы, какие вырезы под какие поля);  
  - **реализации детекции и выравнивания** в коде (сейчас только фиксированные проценты кадра);  
  - **формулы контрольной цифры 731** в явном виде в репо;  
  - **тестового набора фото** с эталонными полями для проверки и подбора параметров.

**Оценка по объёму работ при самостоятельной реализации:**

- **Детекция + перспектива:** 1–3 дня по готовым туториалам/OpenCV (или 2–5 дней в браузере с OpenCV.js/аналогом), если не углубляться в тонкую настройку.
- **Шаблон полей и вырезы:** 1–2 дня на выписывание координат из нормативов или разметку по эталону + код вырезов.
- **Интеграция OCR по зонам и парсеры:** 1–2 дня поверх текущего кода.
- **Подбор параметров и тесты на реальных фото:** от 2–5 дней в зависимости от требований к качеству.
- **Прописка «как у сервиса»:** отдельная задача (разметка или модель под рукопись), ещё несколько дней.

**В сумме:** ориентир **1–2 недели** для одного разработчика до стабильного результата по развороту (ФИО, даты, серия, номер, кем выдан), если целенаправленно добирать недостающее (нормативы/эталон для шаблона, детекция, перспектива) и использовать открытые материалы и референсный код. Полный «аналог Beorg» с пропиской и высокой устойчивостью к ракурсам/качеству фото — заметно больше.

**Резюме:** информации достаточно, чтобы **целенаправленно дописать** пайплайн (детекция → перспектива → регионы по шаблону → OCR → парсинг), но не в виде «всё уже лежит в проекте». Нужно отдельно выписать шаблон полей из нормативов или по эталону, добавить этап детекции/выравнивания и привязать вырезы к документу, а не к кадру. Оценка трудозатрат — порядка 1–2 недель на разворот с приемлемой стабильностью, без учёта прописки «под сервис».
