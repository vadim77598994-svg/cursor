# Специализированные сервисы распознавания паспортов: обзор и применение

## Почему серия и номер не распознаются у нас сейчас

Во внутреннем паспорте РФ **серия (4 цифры) и номер (6 цифр) напечатаны вертикально** на правом краю разворота (стр. 2–3). В Tesseract (и Tesseract.js) **вертикальный текст распознаётся плохо** — это известное ограничение движка. Горизонтальные блоки (ФИО, даты, «Кем выдан») он чаще всего берёт, а вертикальный столбец цифр — нет или с ошибками. Поэтому «вообще не сосканировал» серию/номер — ожидаемое поведение текущего стека, а не только парсера.

**Итог:** надёжно получать серию и номер из фото разворота паспорта РФ с одним только Tesseract.js в браузере практически нельзя без доработок (вырезка зоны, поворот на 90°, отдельный прогон OCR по зоне) и даже тогда результат не гарантирован.

## Сервисы, специализирующиеся на паспортах/документах

### 1. Биорг.Паспорт (beorg.ru) — РФ, паспорта РФ/СНГ

- **Что умеет:** распознавание паспортов РФ, загранпаспортов, водительских, СНИЛС, прописка (в т.ч. рукописная), 50+ типов документов.
- **Интеграция:** REST API, «подключение за 30 минут».
- **Точность:** заявлено 99%+.
- **Деньги:** от ~4 ₽ за распознавание, коммерция от ~5000 ₽/мес; есть бесплатное демо и пилот.
- **Данные:** обработка в РФ, ФЗ-152, сертификаты ФСТЭК, реестр российских программ; есть вариант **on-premise** (в контуре организации).
- **Как применить у нас:** фронт или бэкенд отправляет фото разворота (и при необходимости прописки) в API → получает структурированный JSON (ФИО, дата рождения, серия, номер, кем выдан, дата выдачи, адрес) → подставляем в форму. Можно заменить текущий OCR полностью или использовать только для разворота (серия/номер + при желании все поля).

### 2. Regula Document Reader (regulaforensics.com)

- **Что умеет:** тысячи типов документов, в т.ч. паспорта РФ; MRZ (загранпаспорта) + **VizOCR** — чтение визуальной зоны документов РФ/Беларуси/Украины (как раз разворот с серией/номером).
- **Интеграция:** Web API (REST), есть JavaScript, мобильные SDK.
- **Деньги:** бесплатный пробный период (например, 30 дней), далее коммерческая лицензия.
- **Как применить у нас:** отправка фото в Web API → получение полей документа (в т.ч. серия, номер) → подстановка в нашу форму. Подходит, если нужна именно «документ-ориентированная» распознавалка с поддержкой РФ.

### 3. Dbrain (dbrain.io) — паспорт РФ по ИИ

- **Что умеет:** распознавание паспорта РФ, заявлено 12 полей, 2–3 сек, точность до 99%.
- **Интеграция:** API (нужно уточнять формат на сайте).
- **Как применить у нас:** аналогично — фото → API → JSON с полями → заполнение формы.

### 4. DaData (dadata.ru)

- **Что умеет:** **не OCR**, а проверка и стандартизация уже введённых серии и номера (формат, база недействительных паспортов МВД).
- **Как применить у нас:** после того как серия и номер получены (любым способом — ручной ввод или сторонний OCR), отправляем их в DaData для проверки/валидации. Дополняет любой вариант распознавания.

### 5. Зарубежные (Anyline, Klippa, Scandit и т.п.)

- **Фокус:** в основном **MRZ** (загранпаспорта, ID-карты). Внутренний паспорт РФ **не имеет MRZ**, только визуальная зона.
- **Применимость:** подойдут только если у сервиса есть отдельный режим/модель для «визуальной зоны» или внутренних паспортов РФ; нужно уточнять по каждому.

## Варианты применения у нас

### Вариант A. Гибрид: Tesseract для части полей + API только для серии/номера

- **Идея:** ФИО, даты, «Кем выдан» по-прежнему забираем Tesseract.js + наш парсер (они у нас уже лучше работают). Серию и номер не ждём от Tesseract; вместо этого вырезаем на фронте или бэкенде **правую вертикальную полосу** разворота (где серия/номер), поворачиваем на 90° и либо отправляем **только этот кроп** в дешёвый OCR-API (если такой найдётся), либо отправляем **весь разворот** в специализированный API, но используем из ответа **только** серию и номер, остальное подставляем из Tesseract.
- **Плюсы:** минимум отправки данных в сторонний сервис (можно только кроп), остальное остаётся у текущего стека.
- **Минусы:** два источника правды, нужна аккуратная склейка; кроп и поворот на фронте требуют доработки; не все API отдают «только серию/номер» по кропу.

### Вариант B. Полная замена распознавания разворота на один API

- **Идея:** после съёмки разворота (стр. 2–3) фронт отправляет фото на наш бэкенд; бэкенд вызывает **Биорг.Паспорт** или **Regula** (или Dbrain) и получает все поля разворота: ФИО, дата рождения, серия, номер, кем выдан, дата выдачи. Ответ маппим в нашу форму и отдаём на фронт. Прописку по-прежнему можно обрабатывать вторым шагом (второе фото → тот же API или Tesseract только для адреса).
- **Плюсы:** одна модель под паспорт РФ, стабильные серия/номер и остальные поля, меньше кастомной логики парсера.
- **Минусы:** фото паспорта уходит на сторонний сервис (важно для ФЗ-152 и политики клиники); стоимость (подписка или за запрос); зависимость от API и сети.

### Вариант C. Онлайн только для «сложных» полей, остальное — как сейчас

- **Идея:** по умолчанию работаем как сейчас (Tesseract + два шага + парсер). Если после распознавания **серия или номер пустые** (или пользователь нажал «Не распозналось»), показываем кнопку «Распознать через сервис»: отправка фото в Биорг/Regula/Dbrain, подстановка недостающих полей (или всех) в форму.
- **Плюсы:** данные в сторонний сервис только по желанию или при сбое; экономия запросов.
- **Минусы:** два сценария в коде и в UX; пользователь должен понимать, когда нажимать.

### Вариант D. Валидация через DaData после любого OCR

- **Идея:** не заменяет распознавание, а дополняет. После того как серия и номер получены (Tesseract, API или ручной ввод), наш бэкенд или фронт вызывает DaData API проверки паспорта: формат и не в базе ли недействительных. Показываем пользователю «Паспорт проверен» или «Проверьте серию и номер».
- **Плюсы:** повышение доверия к данным, отсев опечаток и недействительных паспортов.
- **Минусы:** не решает проблему «вообще не сосканировал» — только проверяет уже введённое.

## Рекомендация (технический вывод)

1. **Почему серия/номер не сканируются:** из-за **вертикального расположения** цифр в паспорте РФ и ограничений Tesseract по вертикальному тексту. Это ограничение текущего стека, а не только парсера.

2. **Для стабильного получения серии и номера** разумно использовать **специализированный сервис** под паспорта РФ:
   - **Биорг.Паспорт** — удобен по цене, РФ, API, опция on-premise, под наши сценарии (разворот + прописка).
   - **Regula** — сильный вариант, если нужна поддержка многих типов документов и VizOCR по РФ.
   - **Dbrain** — если достаточно «только паспорт РФ» и простой API.

3. **Как применить:**
   - **Быстрый и предсказуемый вариант:** **B** — полная замена распознавания разворота на один вызов API (Биорг/Regula/Dbrain), прописка — вторым запросом или тем же API. Фронт: два шага съёмки без изменений; после фото разворота отправляем его на бэкенд → бэкенд вызывает выбранный API → возвращаем JSON в форму.
   - **Экономия и контроль данных:** **C** — оставить Tesseract по умолчанию, при пустых серии/номере (или по кнопке) отправлять фото в API и подставлять результат.
   - **В любом случае:** добавить **D** (DaData) для проверки серии/номера после заполнения — дёшево и полезно для качества данных.

4. **Что сделать в коде:**  
   - Ввести на бэкенде конфигурируемый вызов выбранного провайдера (например, `POST /api/v1/passport/recognize` с телом = multipart image; внутри — вызов Биорг/Regula/Dbrain по ключу из env).  
   - На фронте: при двухшаговом сценарии после выбора фото разворота вызывать этот endpoint вместо (или в дополнение к) Tesseract.js; маппить ответ API в те же поля, что и сейчас (patient_fio, passport_series, passport_number, …).  
   - Опционально: кнопка «Распознать через сервис» при пустых серии/номере или после неудачного Tesseract.

Дальше можно зафиксировать один сервис (например, Биорг.Паспорт), получить тестовый доступ и спроектировать точный контракт API (поля запроса/ответа) под наш бэкенд и фронт.
