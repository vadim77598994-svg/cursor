# Если деплой падает (CRASHED) — что проверить и идеи автоотладки

## Что сделано в коде для устойчивости

- **Роутер passport** подключается в `main.py` в `try/except`: при ошибке импорта (например, нет `httpx`) приложение всё равно стартует, эндпоинт `/passport/recognize` просто не регистрируется.
- **Импорт Beorg** перенесён внутрь обработчика: модуль `beorg_recognize` и зависимость `httpx` загружаются только при первом вызове `POST /passport/recognize`, а не при старте приложения.
- **runtime.txt** в `dogovor/backend`: указана версия Python 3.12 (как в Dockerfile), чтобы при сборке через Nixpacks не подтягивалась 3.13 с возможными несовместимостями.

## Что проверить вручную при CRASHED

1. **Railway → сервис бэкенда → Deployments → логи (Build и Deploy/Runtime).**
   - В логах **Build**: ошибки `pip install` (нет пакета, несовместимая версия).
   - В логах **Runtime**: последние строки перед падением — часто видно `ModuleNotFoundError`, `ImportError` или traceback при старте uvicorn.

2. **Root Directory в настройках сервиса.**  
   Должен быть путь к каталогу с `app` и `requirements.txt`, например: `dogovor/backend` (если репозиторий — корень проекта).

3. **Сборка через Docker.**  
   Если в сервисе указан Dockerfile, убедиться, что **Build Path** или **Root Directory** ведут в каталог, где лежит этот Dockerfile (например, `dogovor/backend`).

4. **Переменные окружения.**  
   Обязательные для работы (Supabase, Resend и т.д.) должны быть заданы в Railway; при отсутствии части переменных приложение может падать при инициализации (например, при первом обращении к БД).

## Идеи «автоотладки» при падении (предложение, не реализовано)

Ниже варианты, которые можно внедрить по желанию.

### 1. Health-check с проверкой импортов

- В `GET /health` добавить опциональный режим, например `?heavy=1`, который при старте проверяет импорт тяжёлых/опциональных модулей (`beorg_recognize`, `xhtml2pdf`, и т.д.) и возвращает в JSON список успешных/неуспешных импортов.
- **Плюс:** по одному запросу к работающему инстансу видно, какой модуль не загружается.  
- **Минус:** не помогает, если процесс падает до того, как успевает ответить на запрос.

### 2. Скрипт проверки окружения перед стартом (в Docker/CI)

- В репозитории скрипт (например, `dogovor/backend/check_env.py`), который:
  - импортирует `app.main:app` (или только `app.config`);
  - опционально проверяет наличие ключевых переменных окружения;
  - импортирует опциональные модули (passport, pdf_render) и ловит исключения.
- Запускать его в **Dockerfile** или в **Railway build command** перед `uvicorn` (например, `python check_env.py && uvicorn ...`).
- **Плюс:** падение на этапе сборки/старта с понятной ошибкой в логах Railway.  
- **Минус:** чуть усложняет образ/сборку.

### 3. Логирование при старте в файл / в stderr

- В `main.py` в блоке `lifespan` или при импорте логировать список загруженных роутеров и ключевых модулей (например, «passport: ok», «pdf_render: ok»).
- **Плюс:** в логах Railway сразу видно, на каком шаге запуск остановился.  
- **Минус:** не «чинит» падение, только облегчает разбор.

### 4. Отдельный «минимальный» режим старта

- Переменная окружения, например `MINIMAL_START=1`: приложение поднимается только с `GET /health` и без роутеров contract/passport и без тяжёлых зависимостей.
- **Плюс:** при проблемах с контрактами или Beorg деплой хотя бы поднимается, по `/health` можно убедиться, что платформа и сеть в порядке.  
- **Минус:** нужна дисциплина: на проде этот флаг не включать.

### 5. Railway: автоматический перезапуск и уведомления

- В Railway включить **Restart Policy** (если есть), чтобы контейнер перезапускался после падения.
- Настроить **уведомления** (Slack, Telegram, email) при failed deployment — чтобы сразу видеть, что деплой упал, и открывать логи.

Рекомендация для первого шага: **п. 3** (логирование при старте) и **п. 5** (рестарт + уведомления в Railway). При необходимости добавить **п. 1** (расширенный `/health`) для диагностики на уже поднятом инстансе.
