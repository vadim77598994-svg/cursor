# Варианты генерации PDF для договоров на Railway

Текущее состояние: шаблон `contract.html` (Jinja2) есть, конвертация в PDF отключена из‑за зависимости xhtml2pdf → pycairo → системные библиотеки cairo/pango, которых нет в стандартной сборке Railway.

---

## Сравнение вариантов

| Вариант | Стабильность | Бюджет | Скорость (сборка/запрос) | Сложность |
|--------|---------------|--------|---------------------------|-----------|
| **1. Docker + cairo + xhtml2pdf** | Высокая | Без доп. затрат | Сборка +1–2 мин, запрос как сейчас | Низкая |
| **2. Reportlab (чистый Python)** | Высокая | Без доп. затрат | Сборка быстрая, запрос быстрый | Высокая |
| **3. Внешний API (PDFShift и т.п.)** | Зависит от сервиса | Бесплатный тир, потом платно | Сетевой запрос + задержка | Средняя |
| **4. WeasyPrint** | Высокая | Без доп. затрат | Как xhtml2pdf — тоже нужен cairo | Низкая, но та же проблема |

---

## 1. Docker + cairo + xhtml2pdf (рекомендуется)

**Суть:** Собирать образ по Dockerfile: ставим cairo/pango, потом `pip install` с xhtml2pdf. Код и HTML-шаблон не меняем.

- **Стабильность:** Один раз настроенный образ даёт предсказуемую среду; xhtml2pdf давно используется, баги известны.
- **Бюджет:** Доп. платы нет, тот же Railway; образ чуть больше по размеру.
- **Скорость:** Сборка дольше на 1–2 минуты (установка системных пакетов), время ответа API без изменений.
- **Плюсы:** Минимум правок (Dockerfile + вернуть xhtml2pdf в requirements), сохраняем текущий workflow: Jinja2 → HTML → PDF.
- **Минусы:** Нужно поддерживать Dockerfile и базовый образ (обновления раз в полгода–год).

**Итог:** Оптимальный баланс стабильности, бюджета и скорости при минимальных изменениях.

**Как включить:** В репозитории добавлены `dogovor/backend/Dockerfile` и снова подключён xhtml2pdf в `requirements.txt`. В Railway у сервиса бэкенда укажи **Root Directory** = `dogovor/backend` и включи сборку по **Dockerfile** (в настройках сборки выбери использование Dockerfile, если по умолчанию используется Nixpacks). После деплоя проверь `/health?pdf_check=true`: должно быть `xhtml2pdf_available: true`.

## 2. Reportlab (чистый Python)

**Суть:** Генерировать PDF программно (координаты, шрифты, абзацы), без HTML.

- **Стабильность:** Очень высокая, только Python, без системных библиотек.
- **Бюджет:** Бесплатно.
- **Скорость:** Быстрая сборка и быстрый рендер.
- **Минусы:** Нужно заново описать макет договора в коде (аналог текущего contract.html). Любое изменение текста/полей — правка кода; синхронизация с .docx/дизайном сложнее.
- **Итог:** Имеет смысл, если не хочется использовать Docker; ценой станет перенос и поддержка макета в коде.

---

## 3. Внешний API (PDFShift, DocRaptor, html2pdf.app)

**Суть:** Отправлять готовый HTML на сторонний сервис, получать PDF.

- **Стабильность:** Зависит от доступности и SLA сервиса; ещё один внешний компонент.
- **Бюджет:** Обычно есть бесплатный лимит (десятки–сотни PDF/мес), дальше подписка.
- **Скорость:** Доп. задержка на HTTP-запрос (сотни ms – пара секунд).
- **Минусы:** Зависимость от провайдера, возможные вопросы по передаче персональных данных (HTML договора уходит на их сервер).
- **Итог:** Удобно для прототипа или малого объёма; для продакшена с персональными данными — менее предпочтительно.

---

## 4. WeasyPrint

**Суть:** HTML/CSS → PDF, как xhtml2pdf, но с более современным CSS.

- **Стабильность:** Нормальная при наличии cairo/pango.
- **Бюджет:** Бесплатно.
- **Минус:** Тоже требует cairo/pango в системе — на Railway без Docker та же проблема, что и с xhtml2pdf. С Docker — по сути альтернатива xhtml2pdf при желании сменить движок.
- **Итог:** Не снимает необходимость Docker/системных библиотек; смена с xhtml2pdf на WeasyPrint — отдельная задача без выигрыша по бюджету и простоте деплоя.

---

## Рекомендация

**Вариант 1 (Docker + cairo + xhtml2pdf):**  
Максимально стабильно при нулевом доп. бюджете и без переписывания логики. Единственное изменение — сборка через Dockerfile с установкой cairo; скорость ответа не страдает. Дальше в коде достаточно вернуть xhtml2pdf в `requirements.txt` и при необходимости обновлять базовый образ.

Если по политике нельзя использовать Docker на Railway — тогда вариант 2 (Reportlab) как запасной: стабильно и без системных зависимостей, но с переносом макета договора в код и его поддержкой.
