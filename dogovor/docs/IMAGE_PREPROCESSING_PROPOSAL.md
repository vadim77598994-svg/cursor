# Предобработка фото перед отправкой в Биорг

## Идея

Биорг умеет улучшать качество сам, но вы хотите **сначала попробовать свою предобработку** перед вызовом API — чтобы стабилизировать распознавание разворотов без включения платной/доп. функции у Биорг.

**Цепочка сейчас:** фронт (File) → бэкенд (`spread_bytes`) → base64 → Биорг.  
**Цепочка с предобработкой:** то же, но перед отправкой в Биорг к изображению применяется улучшение (на фронте или на бэкенде).

---

## Варианты реализации

### Вариант A: предобработка на бэкенде (Python)

**Где:** в `passport.py` после `_read_upload_in_memory()`, перед вызовом `recognize_passport()` — передаём в Биорг уже обработанные `bytes`.

**Инструменты:** Pillow (PIL) или OpenCV.

- **Pillow** — лёгкий, обычно уже есть в проектах с генерацией PDF/картинок. Хватает для:
  - ресайз до разумного размера (например, длинная сторона 1600–2000 px);
  - контраст (нормализация гистограммы, при желании CLAHE через `ImageEnhance` или пересчёт каналов);
  - лёгкая резкость (фильтр ImageFilter.SHARPEN / UnsharpMask);
  - приведение к sRGB/одному цветовому пространству.
- **OpenCV** — если позже понадобится:
  - выравнивание документа (детекция контура, перспектива);
  - более продвинутая бинаризация/морфология под OCR.

**Плюсы:** один раз настроили — все клиенты (iOS, Android, браузер) получают одинаковое улучшение; полный контроль над форматом и размером перед Биорг.  
**Минусы:** доп. зависимость (Pillow чаще всего уже есть), +50–200 ms на запрос.

---

### Вариант B: предобработка на фронте (браузер)

**Где:** в `StepScan.tsx` перед вызовом `recognizePassport(file)` — из `File` рисуем в `<canvas>`, применяем операции, получаем новый `Blob`/`File` и его отправляем.

**Инструменты:** только Canvas 2D API или лёгкая библиотека (например, pintura, browser-image-compression или свой мини-пайп).

- Ресайз (масштабирование при рисовании в canvas).
- Контраст/яркость через `getImageData` / `putImageData` (линейная формула по каналам).
- Резкость — свёртка ядром 3×3 по данным пикселей.
- Без тяжёлых библиотек: без полноценного CLAHE и без детекции контура/deskew в браузере без доп. либ.

**Плюсы:** не нагружаем бэкенд, можно показать пользователю превью «улучшенного» кадра.  
**Минусы:** разное качество на разных устройствах, ограниченный набор операций без либ, нужно аккуратно обрабатывать большие фото (память, таймауты).

---

## Что именно улучшает распознавание разворотов

| Что делаем | Зачем | Ожидаемый эффект |
|------------|--------|-------------------|
| **Ресайз** | Привести к 1500–2000 px по длинной стороне | Стабильный размер для Биорг, меньше шума от слишком больших фото, быстрее загрузка. |
| **Контраст** | Вытянуть яркости (нормализация гистограммы или лёгкий CLAHE) | Текст и печать лучше отделяются от фона, особенно при плохом освещении. |
| **Резкость** | Лёгкий unsharp mask / sharpen | Чётче границы букв и цифр — меньше путаницы символов. |
| **Выравнивание (deskew)** | Опционально, при наличии контура документа | Меньше искажений для OCR; сложнее без OpenCV, на первом этапе можно пропустить. |

Сначала разумно внедрить **ресайз + контраст + резкость** и замерить, как часто Биорг начинает стабильно проходить с первого раза.

## Рекомендация по проекту

1. **Стартовать с бэкенда (Вариант A)** — один контур для всех клиентов, проще A/B тестировать (включил предобработку в конфиге — получил единое поведение). Зависимость: **Pillow** (если ещё нет).
2. **Пайп на бэкенде:**  
   декодировать загруженное изображение → ресайз (длинная сторона ≤ 2000 px, сохраняя пропорции) → повышение контраста (например, нормализация или лёгкий `ImageEnhance.Contrast`) → лёгкая резкость → кодировать в JPEG (качество 92–95) или PNG и передать эти `bytes` в `recognize_passport()`.
3. **Оценка эффекта:**  
   - если сейчас часть фото «не проходят» из‑за темноты, размытости или слишком большого размера — предобработка может заметно снизить число повторов и ручного ввода;  
   - если отказы в основном из‑за ракурса/обрезания/не того разворота — выигрыш будет меньше, тут уже важнее подсказки в интерфейсе и, при необходимости, deskew (например, через OpenCV позже).
4. **Фронт (Вариант B)** имеет смысл добавлять как доп. опцию: например, «улучшить перед отправкой» галочкой и превью только на клиенте, если не хотите трогать бэкенд или хотите разгрузить его. Но для единообразия и контроля качества первым шагом логичнее бэкенд.

## Итог

- **Реализовать свою предобработку перед Биорг — да, это реалистично и полезно.**
- **Способ:** начать с **бэкенда на Pillow**: ресайз + контраст + резкость; при необходимости позже добавить выравнивание (OpenCV) или дублирующий/альтернативный контур на фронте.
- **Польза для проекта:** более стабильное срабатывание Биорг на фото с телефона (свет, качество камеры), меньше повторных попыток и переключений на ручной ввод.

Если решите идти в сторону бэкенда — можно следующим шагом набросать конкретный модуль `image_enhance.py` и врезку в `passport.py` с флагом включения (например, через конфиг/переменную окружения).

## Реализация (сделано)

- **Модуль:** `backend/app/services/image_enhance.py` — функция `enhance_for_ocr(image_bytes)`: ресайз (длинная сторона ≤ 2000 px), контраст (×1.25), резкость (UnsharpMask), вывод в JPEG quality 92.
- **Конфиг:** `PREPROCESS_PASSPORT_IMAGE` (bool, по умолчанию `true`) в `app/config.py`.
- **Врезка:** в `passport.py` перед вызовом `recognize_passport()` оба снимка (разворот и прописка) при включённом флаге проходят через `enhance_for_ocr()`; при ошибке предобработки в Биорг уходят исходные байты.

**Отключить для теста:** в `.env` или переменных окружения задать `PREPROCESS_PASSPORT_IMAGE=false`.
