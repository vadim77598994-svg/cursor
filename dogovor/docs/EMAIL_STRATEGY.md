# Стратегия отправки писем: Railway сейчас → Timeweb потом

## Анализ

**Сейчас (Railway):** На планах Free/Hobby исходящий SMTP (порты 465, 587) заблокирован. Яндекс SMTP с Railway недоступен («Network is unreachable»). Варианты: апгрейд на Pro (платно) или отправка через HTTP API.

**Потом (Timeweb):** На российском хостинге исходящий SMTP обычно не режут. Текущий SMTP (Яндекс) будет работать без смены логики; можно оставить только SMTP и не использовать Resend.

**Ограничения:** Письма с персональными данными и вложениями — желательно минимизировать передачу через третьи стороны. На Timeweb отправка напрямую через свой SMTP (Яндекс) предпочтительнее.

---

## Рекомендация

1. **Сделать отправку с выбором провайдера по конфигу**  
   Один и тот же код: либо SMTP (Яндекс), либо Resend (HTTP API). Выбор по переменной окружения, без дублирования бизнес-логики.

2. **На Railway**  
   Использовать **Resend** (`EMAIL_PROVIDER=resend`, `RESEND_API_KEY=...`). Бесплатный тир (порядка 100 писем/день), не зависит от блокировки SMTP. Домен отправителя можно верифицировать в Resend или временно отправлять с их домена.

3. **На Timeweb после переезда**  
   Переключить на **SMTP** (`EMAIL_PROVIDER=smtp`, задать `SMTP_*` как сейчас). Тот же Yandex-ящик, без Resend и без лишней передачи данных через зарубежный API.

4. **Не апгрейтить Railway до Pro** только ради SMTP: переезд на Timeweb планируется, смысла платить за временное решение нет.

---

## Итог

- **Реализация:** в бэкенде один пункт отправки писем с договором, два канала: SMTP и Resend. Выбор по `EMAIL_PROVIDER`.
- **Сейчас:** Railway + Resend — письма доходят.
- **После переезда:** Timeweb + SMTP (Яндекс) — меняются только переменные окружения, код трогать не нужно.
