# Безопасность при запуске проекта

Краткий чеклист перед выносом приложения в прод или доступ по сети.

## Секреты и переменные окружения

- **Не коммитить** `.env` в git (уже в `.gitignore`). Все секреты — только в переменных окружения на сервере/хостинге.
- **Supabase:** использовать `service_role` ключ только на бэкенде, никогда не отдавать его на фронт. Для браузера — только `anon` key, если понадобится прямой доступ к Supabase.
- **SMTP:** пароль от почты хранить только в env (`SMTP_PASSWORD`). На проде задавать через настройки хостинга (Vercel, Railway, и т.д.), не в коде.
- На проде использовать **отдельный проект Supabase** (или отдельную БД), чтобы тестовые данные не смешивались с боевыми.

## Сеть и доступ

- Бэкенд должен быть доступен только по **HTTPS** в проде. Иначе логины, ключи и PDF могут уйти по открытому каналу.
- **CORS:** на проде задать в env переменную `CORS_ORIGINS` — URL фронта через запятую (например `https://dogovor.pyeoptics.com`). См. `docs/DEPLOY.md`.
- Если приложение будет доступно из интернета — продумать **ограничение доступа** к API (по IP, VPN, или простая авторизация для планшетов кабинетов).

## Хостинг

- Фронт (Next.js) и бэкенд (FastAPI) при деплое получают переменные из **секретов хостинга**, не из файла `.env` в репозитории.
- Файлы в Supabase Storage (PDF, подписи) уже отдаются по публичным URL; при необходимости можно перевести бакет в приватный и выдавать signed URL.

## Данные

- Персональные данные (ФИО, паспорт, email) хранятся в Supabase и в письмах. Убедиться, что политика конфиденциальности и согласие на обработку данных учтены (юридически и в интерфейсе).
- Регулярные бэкапы БД — через Supabase Dashboard или свой скрипт.

При добавлении авторизации, 2FA или интеграции с CRM — план безопасности расширить (отдельный документ или раздел).
