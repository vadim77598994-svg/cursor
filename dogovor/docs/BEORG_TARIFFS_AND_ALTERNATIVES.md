## Что значит тариф «Биорг.Разметка датасетов»

Это **не** сервис распознавания паспортов для вашего приложения.

**Разметка датасетов** (data annotation / labeling) — это услуга по **разметке данных для обучения ML-моделей**:

- На изображениях или текстах размечают объекты: прямоугольники (bbox), полигоны, маски.
- Результат — датасет с метками (например, «здесь серия паспорта», «здесь фото»), в форматах вроде COCO.
- Такие датасеты нужны, чтобы **обучить или дообучить** свои модели распознавания (например, детекцию полей паспорта).

**В вашей задаче** вы не обучаете свою модель, а хотите вызывать готовый API и получать поля. Тариф «Разметка датасетов» полезен только если будете **сами делать** свою модель распознавания и нужны размеченные данные. Для подключения Beorg к форме договора он не нужен.

**Облачное vs локальное решение: что подходит проекту**

| Критерий | Облако (Beorg, Bicore, Dbrain и др.) | Локальное (Smart Engines, PassportReader 2.0, Cognitive Passport API) |
|----------|--------------------------------------|----------------------------------------------------------------------|
| **Развёртывание** | Не нужны серверы в клиниках, только доступ в интернет | Нужен сервер или ПК в каждом контуре (или один общий сервер у вас) |
| **Паспортные данные** | Фото уходят к провайдеру API; можно не сохранять у себя после ответа | Данные не покидают ваш контур — плюс для 152‑ФЗ и политики «не хранить паспорта» |
| **Стоимость** | Помесячно/за скан; предсказуемо при известном трафике | Лицензия/SDK разово или по серверам; свои затраты на поддержку и хостинг |
| **Обновления моделей** | Провайдер обновляет сам | Обновления ставятся у вас (или не ставятся) |
| **Интеграция** | REST API из любого места (планшет, браузер → ваш бэкенд → облако) | Нужен свой сервис, который дергает SDK/локальный API; из веб-приложения — только через ваш бэкенд |
| **Офлайн/закрытый контур** | Не работает без интернета | Может работать в изолированной сети клиники |
| **Масштаб** | Легко масштабировать по количеству клиник без своей инфраструктуры | Масштаб = больше серверов/лицензий и администрирования |

**Плюсы облака для Пай Оптикс:** клиники без своего IT, планшеты/телефоны, один бэкенд (например на Railway); не нужно разворачивать и сопровождать серверы распознавания; быстрый старт. **Минусы:** передача фото третьей стороне (выбирать провайдера с договором и политикой обработки ПДн); зависимость от доступности API.

**Плюсы локального варианта:** паспортные данные не передаются наружу; при желании можно вообще не хранить фото (обработка «на лету»). **Минусы:** нужна инфраструктура (сервер/ПК с SDK или HTTP-сервисом), обновления и администрирование на вас; не все SDK дают удобный REST API из коробки (Cognitive Passport API — DLL/ActiveX; PassportReader 2.0 — HTTP API-сервис, ближе к нашему сценарию).

**Вывод для проекта:** при отсутствии своего серверного контура в клиниках и желании не хранить паспортные данные оптимален **облачный API** с политикой «не сохраняем фото после ответа» (как у Биорг). Локальное имело бы смысл при строгом требовании «никакой передачи ПДн третьим лицам» и готовности поднимать и содержать свой сервис распознавания (например на базе PassportReader 2.0 или Smart Engines).

**Оценка для проекта договоров Пай Оптикс:**

1. **Биорг.Универсальный (20 000 ₽/мес)** — наиболее прямой вариант: один тариф с разворотом и пропиской, понятная цена, API уже интегрирован в проект, при желании можно не хранить фото (только вызов и подстановка в договор).
2. **Bicore (0,50 ₽/скан)** — российский сервис с очень низкой трафикой; имеет смысл запросить демо и уточнить наличие прописки и REST API под наш стек.
3. **Dbrain** — разумная альтернатива, если по запросу предложат выгоднее или нужен упор на антифрод; цену и условия нужно уточнять.
4. **Regula** — если нужна максимальная база документов и не важен открытый прайс; обычно дороже и под enterprise.
5. **Smart Engines** — если важна только обработка в своём контуре (без облака): SDK, развёртывание у себя, без передачи фото третьим лицам; нет готового облачного API.

Итого: для «разворот + прописка в договор» оптимален **Биорг.Универсальный**; «Разметка датасетов» для этой задачи не используется — это про разметку данных для обучения своих моделей.
