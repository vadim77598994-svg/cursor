# Анализ сервисов/инструментов для проекта «Договор»

Оценка: насколько каждый пункт может улучшить проект, когда к нему вернуться, и что можно сделать уже сейчас стандартными средствами.

---

## 1. Clean architecture enforcer

**Что может дать:** проверки/правила, чтобы код не смешивал слои (домен, use cases, адаптеры). В проде — предсказуемая структура, проще онбординг и рефакторинг.

**Обоснование:** Сейчас бэкенд небольшой (API → сервисы → БД/Storage). Жёсткий clean architecture для текущего размера может быть избыточен; полезнее станет, когда появятся несколько доменов (договоры, CRM, отчёты) или в команде будет больше одного разработчика.

**Когда вернуться:** после стабилизации прод-запуска и первых доработок по фидбеку; перед масштабированием или интеграцией с CRM.

**Установка:** если это плагин/расширение в Cursor/IDE — ставится вручную. Универсальная альтернатива: зафиксировать структуру папок в `docs/` и добавить линтер-правила (например, запрет импортов из `app.api` в `app.models`), когда понадобится.

---

## 2. Auto refactor pro

**Что может дать:** автоматическое переименование, выделение функций, приведение к единому стилю без ручного прохода по файлам.

**Обоснование:** Полезно при рефакторинге (разбить большой файл, переименовать сущности). Сейчас код ещё не разросся, критичной необходимости нет; станет актуальнее при доработках UI и добавлении модулей.

**Когда вернуться:** когда начнёте целенаправленно упрощать/переструктурировать код (например, перед добавлением тестов или CRM).

**Установка:** зависит от продукта (IDE/плагин). В Cursor/VS Code встроенный Refactor (F2, Extract Method и т.д.) уже даёт базовый авторефакторинг.

---

## 3. Smart test generator

**Что может дать:** генерация заготовок юнит- и интеграционных тестов по коду или по сценариям. Покрытие критичных путей (генерация договора, отправка email, API) без полного ручного написания.

**Обоснование:** Сейчас тестов нет. Даже минимальное покрытие (health, генерация контракта с моками) снизит риск поломок при деплое и рефакторинге. Генератор тестов даёт максимальный эффект, когда уже есть каркас (pytest для бэка, Jest/React Testing Library для фронта).

**Когда вернуться:** сразу после выхода в прод — первым делом добавить каркас тестов и 1–2 ключевых теста; затем подключать генератор для расширения покрытия.

**Установка:** если это отдельный сервис — по инструкции продукта. Стандартный вариант: мы добавляем каркас (pytest, при необходимости Jest), один-два примера тестов — тогда любым «smart test generator» будет проще генерировать релевантный код.

---

## 4. Dependency optimizer

**Что может дать:** проверка уязвимостей и устаревших зависимостей, предложения обновлений, уменьшение дублирования (npm/pip).

**Обоснование:** Меньше рисков безопасности, совместимость с новыми версиями Node/Python. Для проекта с Supabase, SMTP и персональными данными актуально уже сейчас.

**Когда вернуться:** можно включить сразу и держать включённым (регулярные проверки).

**Установка:** часть можно сделать без сторонних сервисов: Dependabot (GitHub), `npm audit` / `pip audit` (или `safety`). Ниже добавлен базовый CI, который может запускать проверки зависимостей.

---

## 5. API contract guardian

**Что может дать:** проверка, что фронт и бэкенд соблюдают один контракт (формат запросов/ответов). Тесты по OpenAPI или Pact: при изменении API контракт-тесты падают и показывают несовместимость.

**Обоснование:** FastAPI уже отдаёт OpenAPI. Контракт-тесты полезны, когда фронт и бэкенд развиваются параллельно или над бэкендом работает другой человек; на текущем этапе один разработчик и один репо — польза есть, но не критична.

**Когда вернуться:** после появления второго разработчика или отдельного фронт/бэкенд репозитория; или перед крупным рефакторингом API.

**Установка:** можно добавить экспорт OpenAPI в репо и простой тест (например, pytest), который дергает `/openapi.json` и проверяет наличие нужных эндпоинтов и полей. Полноценный Pact/Guardian — по документации выбранного сервиса.

---

## 6. Performance scanner

**Что может дать:** замер времени ответа API, размера бандла фронта, Core Web Vitals. Выявление узких мест до того, как пользователи начнут жаловаться.

**Обоснование:** Для планшетов в кабинетах и загрузки Tesseract.js производительность уже важна. Сканер даёт объективные метрики и регрессии после изменений.

**Когда вернуться:** после первого прод-запуска — зафиксировать базовые метрики; затем подключать к CI или раз в спринт.

**Установка:** Lighthouse (ручной запуск или Lighthouse CI в GitHub Actions), бандл-анализатор для Next.js. Часть можно автоматизировать в том же CI, что и тесты.

---

## 7. CI/CD quick setup

**Что может дать:** автоматический прогон тестов и линтеров при push/PR, опционально — деплой на Railway/Vercel по ветке. Меньше шансов задеплоить сломанный код.

**Обоснование:** Сейчас деплой ручной. Даже минимальный CI (lint + build + при появлении — тесты) сразу повышает стабильность. Для продакшена полезно перед каждым деплоем убедиться, что сборка зелёная.

**Когда вернуться:** логично включить до или сразу после первого прод-запуска.

**Установка:** можно сделать сейчас стандартными средствами: GitHub Actions для бэкенда (lint/install) и фронта (build). Ниже добавлен пример workflow.

---

## 8. Context memory sync

**Что может дать:** зависит от продукта. Варианты: синхронизация контекста между сессиями ИИ, между устройствами, или «память» проекта для ассистента. Может уменьшить повторение одних и тех же инструкций.

**Обоснование:** Улучшает работу с ассистентом/ИИ, а не напрямую качество кода. Полезно при длительной поддержке проекта и множестве правок.

**Когда вернуться:** когда накопится много контекста (документация, решения, баги) и станет заметно, что ассистент «забывает» важное. Или при переходе на инструмент с явной «памятью проекта».

**Установка:** по инструкции конкретного сервиса (часто это настройки Cursor/IDE или отдельное приложение). В репо можно вести `docs/` и `CHANGELOG` — это уже базовая «память» проекта.

## Что добавлено в репозиторий без сторонних сервисов

- **CI (GitHub Actions):** файл `.github/workflows/dogovor-ci.yml` — при push в `main`/`master` и при PR запускается:
  - **Backend:** установка зависимостей, `pytest tests/`, базовая проверка синтаксиса.
  - **Frontend:** `npm ci`, `npm run build`.
- **Каркас тестов бэкенда:** папка `dogovor/backend/tests/`, тест `test_health.py` (проверка `/health`). Так уже есть точка для расширения тестами и для «smart test generator».
- **Рекомендация по зависимостям:** раз в период запускать `npm audit` (фронт) и проверку pip/`safety` (бэкенд); в GitHub при желании включить Dependabot (Settings → Security → Dependabot).

Конкретные коммерческие продукты из списка (если это плагины или SaaS) устанавливаются по их документации; выше — когда и зачем к ним возвращаться и чем можно усилить проект уже сейчас.
