# Часть 1. Бэкенд на Railway — подробный план

Разделение: что уже сделано в репозитории, что делаешь ты вручную, где возможны ошибки и как проверить.

---

## Общая последовательность

1. **В репозитории (сделано / сделаю я)** — подготовка к деплою: конфиги, Procfile, проверка зависимостей.
2. **Твои действия** — регистрация в Railway, подключение GitHub, настройка сервиса и переменных.
3. **Проверка** — открыть `/health`, убедиться, что бэкенд отвечает по HTTPS.

## Railway: платный или бесплатный?

- **Бесплатно:** при регистрации дают пробный период с кредитами ($5), без карты. Дальше есть Free-план с очень небольшим месячным кредитом (~$1) — для малого трафика может хватить, но лимиты жёсткие.
- **Платно:** для стабильного продакшена обычно подключают карту и тариф Hobby ($5/мес) или выше. Актуальные лимиты и цены: [railway.app/pricing](https://railway.app/pricing).
- **Альтернативы с бесплатным тиром:** [Render](https://render.com) (free tier с ограничениями), [Fly.io](https://fly.io) (есть free allowance). Шаги деплоя похожи: свой Root Directory, переменные, команда запуска.

## Блок B. Что делаешь ты (по шагам)

### Шаг B.1. Регистрация и создание проекта

1. Открой [railway.app](https://railway.app).
2. Войди через **GitHub** (Login with GitHub).
3. Нажми **New Project**.
4. Выбери **Deploy from GitHub repo**.
5. Если репозиторий с проектом (где есть папка `dogovor`) ещё не подключён — нажми **Configure GitHub App** и выдай Railway доступ к нужному репо.
6. Выбери этот репозиторий и подтверди. Railway создаст сервис и начнёт первый деплой (он может быть красным, пока не заданы Root Directory и переменные — это нормально).

**Где нужна помощь:** если репо приватный, Railway должен иметь к нему доступ. Если репо в организации — могут понадобиться права на установку GitHub App.

---

### Шаг B.2. Root Directory и команда запуска

**Важно:** в интерфейсе Railway ты **не увидишь** дерево папок репозитория. Папка `dogovor` есть только в **GitHub** (в том репо, который ты подключил). В Railway нужно вручную прописать путь к ней в настройках сервиса.

1. В проекте Railway открой созданный **сервис** (карточка сервиса, не проект).
2. Перейди в **Settings** (иконка шестерёнки или вкладка **Settings**).
3. Найди блок **Source** (или **Build** / **Deploy**). В нём должно быть поле:
   - **Root Directory** или **Monorepo Root** или **Service Root**.
4. В это поле введи **ровно:**  
   **`dogovor/backend`**  
   Без слэша в начале, без точки в конце. Так Railway будет собирать и запускать приложение из папки `dogovor/backend` внутри твоего репо (там лежат `requirements.txt`, `app/`, `Procfile`).
5. Сохрани (кнопка **Save** / **Update** или просто выход из поля, если сохраняется автоматически).
4. Найди **Build Command** (если есть). Можно оставить пустым — тогда Railway по умолчанию выполнит что-то вроде `pip install -r requirements.txt`. Или явно задать:  
   `pip install -r requirements.txt`
5. Найди **Start Command** / **Start** (команда запуска). Задай:  
   `uvicorn app.main:app --host 0.0.0.0 --port $PORT`  
   Переменная `$PORT` подставляется Railway сам. Сохрани настройки.

**Где возможна ошибка:** если Root Directory не указан или указан неверно (например, корень репо), сборка будет искать `app/` не там и упадёт. Всегда проверяй, что в логе сборки видно установку из `dogovor/backend` (или что рабочая директория при сборке — `dogovor/backend`).

### Шаг B.3. Переменные окружения (Environment Variables)

1. В том же сервисе открой вкладку **Variables** (или **Environment**).
2. Добавь переменные по одной. Имя — слева, значение — справа. Ниже список того, что должно быть в твоём случае.

**Обязательные (без них приложение не стартует или не работает):**

| Имя | Значение | Откуда взять |
|-----|----------|--------------|
| `SUPABASE_URL` | `https://sojsihvhpvmzzdmqgpri.supabase.co` | Твой проект Supabase → Settings → API → Project URL |
| `SUPABASE_SERVICE_KEY` | Длинный JWT-ключ | Тот же раздел → Project API keys → **service_role** (Reveal) |
| `SUPABASE_STORAGE_BUCKET` | `contracts` | Имя бакета в Supabase Storage |

**Для CORS (обязательно после деплоя фронта):**

| Имя | Значение | Когда заполнять |
|-----|----------|------------------|
| `CORS_ORIGINS` | URL фронта без слэша в конце | Сразу можно оставить пустым или временно `https://твой-проект.vercel.app`; после деплоя фронта (Часть 2) — подставить итоговый URL (или несколько через запятую). |

**Для отправки PDF на email:**

| Имя | Значение |
|-----|----------|
| `SMTP_HOST` | `smtp.yandex.ru` |
| `SMTP_PORT` | `465` |
| `SMTP_USER` | `support@pyeoptics.com` |
| `SMTP_PASSWORD` | Пароль приложения Яндекса (тот же, что в локальном `.env`) |
| `SMTP_FROM_EMAIL` | `support@pyeoptics.com` |
| `SMTP_FROM_NAME` | `Пай Оптикс` |

3. После сохранения переменных Railway перезапустит сервис (пересоберёт при необходимости).

**Где нужна помощь:** значения `SUPABASE_URL` и `SUPABASE_SERVICE_KEY` ты подставляешь сам — они есть в твоём Supabase Dashboard. В репозитории их нет и не должно быть. Если забудешь одну из переменных, в логах Railway будет ошибка при старте (например, про отсутствующий `SUPABASE_SERVICE_KEY`).

---

### Шаг B.4. Публичный URL (домен)

1. В настройках сервиса найди раздел **Networking** / **Public Networking**.
2. Нажми **Generate Domain** (или **Add Domain**). Railway выдаст домен вида `что-то.up.railway.app`.
3. Скопируй полный URL, например:  
   **`https://dogovor-production-xxxx.up.railway.app`**  
   Это и есть **URL бэкенда**. Сохрани его: он понадобится для Части 2 (фронт) и для обновления `CORS_ORIGINS`.

**Где возможна ошибка:** без сгенерированного домена сервис не будет доступен снаружи. После генерации подожди 1–2 минуты и проверь шаг B.5.

---

### Шаг B.5. Проверка

1. В браузере открой: **`https://твой-url/health`** (подставь свой домен из B.4).
2. Ожидаемый ответ: **`{"status":"ok"}`** и код 200.
3. Если видишь ошибку или таймаут:
   - Проверь в Railway вкладку **Deployments** / **Logs**: нет ли падения при старте (например, из-за отсутствующей переменной или ошибки импорта).
   - Убедись, что в Variables нет опечаток в именах (`SUPABASE_URL`, а не `SUPABASE_URI` и т.п.).

**Результат Части 1:** у тебя есть постоянный HTTPS-URL бэкенда, который отвечает на `/health`. Этот URL дальше вставляешь во фронт (Часть 2) и в `CORS_ORIGINS` после деплоя фронта.

## «Не вижу папку dogovor» в Railway

В Railway **нет** экрана с папками репозитория. Работает так:

1. Ты подключаешь **один репозиторий с GitHub** (тот, где у тебя лежит папка `dogovor` — например репо `cursor` или как он у тебя назван).
2. В настройках **сервиса** (не проекта) в поле **Root Directory** ты **вручную вводишь** путь: **`dogovor/backend`**.
3. Railway при сборке клонирует весь репо и дальше работает только внутри указанной папки — как будто бы `dogovor/backend` стала корнем проекта.

Проверь:
- Подключён ли **тот самый** репозиторий, в котором есть папка `dogovor` (посмотри на GitHub: в корне репо должна быть папка `dogovor`, внутри неё `backend`, `frontend`, `docs` и т.д.).
- В **Settings** сервиса в **Root Directory** указано именно **`dogovor/backend`** (без начального `/` и без кавычек).

После сохранения Root Directory Railway перезапустит деплой; в логе сборки будет видно, что команды выполняются из папки с `requirements.txt` и `app/`.

## Блок C. Возможные проблемы

| Симптом | Что проверить |
|--------|----------------|
| Сборка падает с "No module named 'app'" | Root Directory должен быть `dogovor/backend`, не корень репо. |
| При старте ошибка про Supabase | В Variables заданы `SUPABASE_URL` и `SUPABASE_SERVICE_KEY` (именно service_role). |
| При старте ошибка про порт | Start Command должен содержать `--port $PORT` (буквально $PORT). |
| 502 Bad Gateway / таймаут | Подождать 1–2 минуты после первого деплоя; посмотреть логи на предмет падения процесса. |
| CORS при запросах с фронта | После деплоя фронта обновить `CORS_ORIGINS` на полный URL фронта и перезапустить сервис. |
