# Настройка и запуск «Договор» — подробная инструкция

Пошаговая настройка: Supabase (БД + Storage) → бэкенд FastAPI → фронт Next.js. В конце — проверка сценария и типичные ошибки.

---

## Что понадобится

- Аккаунт [Supabase](https://supabase.com) (бесплатный тариф подойдёт).
- На компьютере: **Node.js** (для фронта), **Python 3.10+** (для бэкенда).
- Терминал (командная строка) и любой текстовый редактор для `.env`.

---

## Часть 1. Supabase

Supabase — это облачный сервис: в нём живут **база данных** (таблицы с кабинетами, оптометристами, договорами) и **хранилище файлов** (куда бэкенд кладёт готовые PDF). Сначала создаём проект, потом «объясняем» базе, какие таблицы нужны и какие данные в них сразу заложить. Это и делается через SQL.

---

### 1.1 Вход и создание проекта

1. Откройте в браузере: **https://supabase.com/dashboard**
2. Войдите в аккаунт (Sign in) или зарегистрируйтесь (Sign up) через email или GitHub.
3. После входа вы попадёте на страницу со списком проектов. Слева обычно меню (Project overview, SQL Editor, Table Editor и т.д.), справа — карточки проектов или кнопка **New project**.
4. Нажмите **New project** (или **Create a new project**).
5. Заполните форму:
   - **Name** — любое имя (например `dogovor` или название организации). Это только для отображения в интерфейсе Supabase.
   - **Database Password** — придумайте надёжный пароль и сохраните его. Он нужен для прямого доступа к PostgreSQL (через сторонние клиенты). Для нашего приложения пароль не используется — мы работаем через API и ключи.
   - **Region** — выберите регион поближе к вам (например Frankfurt, если в Европе). От региона зависит скорость ответа базы.
6. Нажмите **Create new project**. Создание занимает 1–2 минуты. Дождитесь, пока статус проекта станет зелёным (Ready).

Если проект уже есть — на главной выберите его из списка, кликнув по названию. После этого вы внутри проекта: в левом меню доступны разделы этого проекта.

### 1.2 Что такое SQL и зачем его запускать

**SQL** (Structured Query Language) — это язык запросов к базе данных. Им описывают команды вроде: «создай таблицу с такими-то колонками», «вставь в неё эти строки», «создай счётчик и функцию для номера договора». Файл **`001_contracts_schema.sql`** — это как раз такой набор команд (скрипт) в виде обычного текста.

В нём по порядку написано примерно следующее:

- **CREATE TABLE** — создать таблицу (например `dogovor_locations` с колонками: название кабинета, адрес, город, префикс номера и т.д.).
- **CREATE SEQUENCE** — создать счётчик, который будет выдавать 1, 2, 3… для уникальных номеров договоров.
- **CREATE FUNCTION** — создать функцию `get_next_contract_number`, которая по префиксу (например R10) возвращает номер вида R10-2025-00001.
- **INSERT INTO** — вставить готовые строки: три кабинета и восемь оптометристов.

Когда вы **выполняете** этот SQL в Supabase (кнопка Run), база по очереди выполняет все команды: появляются таблицы, счётчик, функция и начальные данные. Без этого шага в проекте не будет нужных таблиц, и приложение не сможет ни загрузить список кабинетов, ни сохранить договор. Запуск SQL нужен **один раз** при первой настройке (или при переносе на новый проект).

---

### 1.3 Где взять файл со SQL и как его выполнить

Файл лежит в репозитории проекта по пути:

**`dogovor/supabase/001_contracts_schema.sql`**

То есть в папке `dogovor`, внутри неё папка `supabase`, внутри — файл `001_contracts_schema.sql`. Его можно открыть в Cursor, VS Code или любом текстовом редакторе.

Пошагово:

1. В Supabase в **левом меню** найдите раздел **SQL Editor** (часто с иконкой в виде `</>` или надписи SQL). Нажмите на него.
2. Вверху страницы SQL Editor нажмите **+ New query** (или **New query**). Откроется новая вкладка с пустым полем для ввода текста.
3. Откройте файл **`dogovor/supabase/001_contracts_schema.sql`** в редакторе на своём компьютере. Выделите весь текст (Ctrl+A или Cmd+A), скопируйте (Ctrl+C или Cmd+C).
4. Вернитесь в браузер, в окно запроса в Supabase. Вставьте скопированный текст (Ctrl+V или Cmd+V). Должен появиться весь скрипт: команды `create table`, `insert into` и т.д.
5. Внизу справа нажмите зелёную кнопку **Run** (или используйте Ctrl+Enter / Cmd+Enter). Supabase отправит весь скрипт в базу и выполнит его.
6. Внизу страницы появится результат. Обычно это сообщение вроде **Success. No rows returned** — для команд создания таблиц и вставки данных это нормально (они не возвращают строки, а только меняют структуру и данные).

Если появилась **ошибка** (красное сообщение) — скопируйте текст ошибки. Часто это бывает из‑за того, что скрипт уже выполняли раньше (таблицы уже есть). В таком случае можно либо проигнорировать, если таблицы и данные на месте, либо создать новый проект и выполнить SQL в нём один раз с нуля.

### 1.4 Что именно создаётся этим SQL-скриптом

Кратко по объектам в базе:

| Объект | Назначение |
|--------|------------|
| Таблица **dogovor_locations** | Кабинеты проверки зрения: название, полный адрес, город, префикс номера договора (M08, R10, B01), порядок сортировки. |
| Таблица **dogovor_staff** | Оптометристы: ФИО, город (чтобы подставлять только врачей этого города), позже — ссылка на изображение подписи. |
| Таблица **dogovor_contracts** | Каждый подписанный договор: номер, ссылка на кабинет и врача, ФИО пациента, путь к PDF в Storage, дата. |
| **Sequence** dogovor_contract_seq | Внутренний счётчик 1, 2, 3… для формирования уникального номера договора. |
| Функция **get_next_contract_number(prefix)** | По префиксу (например R10) возвращает следующий номер: R10-2025-00001, R10-2025-00002 и т.д. Вызывается из бэкенда. |
| **RLS-политики** | Правила доступа: кто может читать таблицы и кто может вставлять записи в договоры (для нашего сценария — доступ по service_role ключу). |
| **Начальные данные (INSERT)** | 3 кабинета (Москва Потаповский, СПб Рубинштейна, СПб Бармалеева) и 8 оптометристов (5 в СПб, 3 в Москве). |

### 1.5 Как проверить, что SQL выполнился

1. В левом меню Supabase выберите **Table Editor**.
2. В списке таблиц должны быть: **dogovor_locations**, **dogovor_staff**, **dogovor_contracts** (названия могут отображаться без префикса `public.`).
3. Кликните по **dogovor_locations**. Должны отобразиться 3 строки — три кабинета с названиями и адресами.
4. Откройте **dogovor_staff** — должно быть 8 строк с ФИО оптометристов и городами.

Если таблицы есть и данные на месте — часть 1 с SQL выполнена правильно. Дальше — создать bucket для PDF и взять ключи API.

---

### 1.6 Создание bucket для PDF

1. В левом меню Supabase выберите **Storage**.
2. Нажмите **New bucket**.
3. В поле **Name** введите точно: **contracts** (именно так ожидает бэкенд, либо потом укажите другое имя в `.env`).
4. Опции:
   - **Public bucket** — включите, если нужны публичные ссылки на PDF (например для скачивания по прямой ссылке). Если оставить приватным — доступ только через API с ключами.
   - **File size limit** и **Allowed MIME types** можно не менять для MVP.
5. Нажмите **Create bucket**.

Проверка: в списке buckets должен появиться **contracts**. Внутри пока пусто — файлы появятся после первого успешного создания договора через приложение.

---

### 1.7 Копирование URL и ключа для бэкенда

Бэкенду нужны URL проекта и **service_role** ключ (не anon).

1. В левом меню внизу нажмите **Project Settings** (иконка шестерёнки).
2. Перейдите в раздел **API**.
3. В блоке **Project URL** скопируйте ссылку (вид: `https://xxxxxxxxxxxx.supabase.co`). Это **SUPABASE_URL**.
4. Ниже найдите блок **Project API keys**. Там два ключа:
   - **anon** `public` — для клиентского кода (браузер); для бэкенда не используем.
   - **service_role** `secret` — полный доступ к БД и Storage. Нажмите **Reveal** и скопируйте длинную строку (начинается с `eyJ...`). Это **SUPABASE_SERVICE_KEY**.

Сохраните оба значения в надёжное место (например в заметки). Они понадобятся в следующей части.

## Часть 2. Бэкенд (FastAPI)

Бэкенд отдаёт API: список кабинетов и оптометристов, генерация номера договора, рендер PDF и загрузка в Storage.

### 2.1 Переход в папку и создание .env

В терминале перейдите в папку бэкенда (из корня репозитория):

```bash
cd dogovor/backend
```

Создайте файл `.env` из примера:

```bash
cp .env.example .env
```

Откройте файл `.env` в редакторе. Должно быть примерно так:

```env
# Supabase (тот же проект или отдельный для договоров)
# Dashboard → Project Settings → API: URL и service_role key
SUPABASE_URL=https://xxxxx.supabase.co
SUPABASE_SERVICE_KEY=eyJhbGc...

# Bucket для PDF договоров (создать в Supabase Storage)
STORAGE_BUCKET=contracts
```

Замените значения на свои:

- **SUPABASE_URL** — вставьте скопированный Project URL из Supabase (без слэша в конце).
- **SUPABASE_SERVICE_KEY** — вставьте скопированный **service_role** ключ целиком, без пробелов и кавычек.
- **STORAGE_BUCKET** — оставьте `contracts`, если bucket создали с таким именем; иначе укажите имя вашего bucket.

Сохраните файл. Файл `.env` не должен попадать в git (как правило, он уже в `.gitignore`).

### 2.2 Виртуальное окружение и зависимости (рекомендуется)

Так зависимости проекта не смешаются с системным Python.

**macOS / Linux:**

```bash
cd dogovor/backend
python3 -m venv .venv
source .venv/bin/activate
```

После активации в начале строки терминала появится `(.venv)`.

**Windows (PowerShell):**

```bash
cd dogovor\backend
python -m venv .venv
.venv\Scripts\activate
```

Установка пакетов:

```bash
pip install -r requirements.txt
```

Если появятся ошибки:

- Убедитесь, что используете Python 3.10 или новее: `python3 --version` (или `python --version` на Windows).
- На некоторых системах нужен `pip install --upgrade pip`, затем снова `pip install -r requirements.txt`.

### 2.3 Запуск сервера

В той же папке `dogovor/backend` (с активированным `.venv`, если используете):

```bash
uvicorn app.main:app --reload --port 8000
```

Ожидаемый вывод в терминале:

```
INFO:     Uvicorn running on http://127.0.0.1:8000 (Press CTRL+C to quit)
INFO:     Started reloader process ...
INFO:     Started server process ...
INFO:     Waiting for application startup.
INFO:     Application startup complete.
```

Сервер будет перезапускаться при изменении кода (`--reload`). Окно терминала не закрывайте.

Проверка:

- В браузере откройте: http://localhost:8000/health  
  Должен вернуться ответ: `{"status":"ok"}`.
- Откройте: http://localhost:8000/docs  
  Должна открыться Swagger-документация API (эндпоинты `/api/v1/locations`, `/api/v1/staff`, `/api/v1/contracts/generate`).

Если при запуске видите ошибку про `SUPABASE_URL` или `SUPABASE_SERVICE_KEY` — проверьте, что в `.env` подставлены реальные значения и что файл лежит именно в `dogovor/backend`.

**Распознавание паспорта (Beorg):** чтобы работал сторонний сервис Биорг, в `dogovor/backend/.env` должны быть заданы `BEORG_PROJECT_ID`, `BEORG_TOKEN`, `BEORG_MACHINE_UID`. После изменения `.env` **перезапустите бэкенд** (Ctrl+C и снова `uvicorn ...`), иначе переменные не подхватятся. В логах при старте должно быть: `Passport router loaded; Beorg: configured`. Зависимость `python-multipart` уже в `requirements.txt` (нужна для загрузки фото).

## Часть 3. Фронт (Next.js)

Фронт работает на порту **3001**, чтобы не пересекаться с другим приложением на 3000.

### 3.1 Установка зависимостей и запуск

Откройте **новый** терминал (бэкенд оставьте запущенным в первом). Перейдите в папку фронта:

```bash
cd dogovor/frontend
```

Установите зависимости (один раз):

```bash
npm install
```

Запустите dev-сервер:

```bash
npm run dev
```

В консоли должно появиться что-то вроде:

```
▲ Next.js 15.x.x
- Local:        http://localhost:3001
```

Откройте в браузере: **http://localhost:3001**

Должна открыться страница «Оформление договора» с прогресс-баром и первым шагом «Кабинет и врач». Если вместо списка кабинетов видите сообщение об ошибке — проверьте, что бэкенд запущен на порту 8000 и в следующем пункте при необходимости указан правильный URL API.

---

### 3.2 Переменные окружения фронта (если бэкенд не на localhost:8000)

По умолчанию фронт обращается к `http://localhost:8000`. Если бэкенд на другом адресе или порту (например на другом компьютере в сети), создайте в папке **dogovor/frontend** файл **`.env.local`**:

```env
NEXT_PUBLIC_DOGOVOR_API_URL=http://localhost:8000
```

Замените URL на актуальный (без слэша в конце). После изменения перезапустите `npm run dev`.

## Часть 4. Проверка полного сценария

Цель — пройти весь путь от выбора кабинета до появления номера договора и PDF в Storage.

1. **Шаг 1 — Кабинет и врач**  
   На http://localhost:3001 должны подгрузиться три кабинета. Выберите один (например «Рубинштейна, 10»). Ниже появится список оптометристов этого города — выберите одного. Нажмите **Далее — сканирование паспорта**.

2. **Шаг 2 — Сканирование**  
   Нажмите **Ввести данные вручную** (камера/OCR пока не подключены).

3. **Шаг 3 — Проверка данных**  
   Заполните форму. Поле **ФИО пациента** обязательно. Остальные поля можно заполнить тестовыми данными или оставить пустыми. Нажмите **Далее — подпись**.

4. **Шаг 4 — Подпись**  
   В белом поле «нарисуйте» подпись мышью (или пальцем на тачскрине). При необходимости нажмите **Очистить** и нарисуйте снова. Нажмите **Сгенерировать договор**.

Ожидаемый результат:

- На экране появится сообщение **Договор оформлен** и номер договора (например `R10-2025-00001`).
- Если bucket `contracts` создан и ключи в бэкенде верные — ниже будет строка вида **PDF сохранён: 2025/R10-2025-00001.pdf**.

Проверка в Supabase:

- **Table Editor** → таблица **dogovor_contracts** — должна появиться новая строка с этим номером, `location_id`, `staff_id`, `patient_fio`, `pdf_path`.
- **Storage** → bucket **contracts** → папка по году (например **2025**) — внутри должен быть файл PDF с соответствующим именем.

## Часть 5. Типичные проблемы и решения

| Симптом | Что проверить |
|--------|----------------|
| На шаге «Кабинет и врач» пишет, что не удалось загрузить список кабинетов | Бэкенд запущен на порту 8000? Откройте http://localhost:8000/api/v1/locations в браузере — должен вернуться JSON с массивом. Если нет — проверьте `.env` бэкенда (URL и **service_role** ключ) и что SQL из `001_contracts_schema.sql` выполнен. |
| При запуске бэкенда ошибка `SUPABASE_URL and SUPABASE_SERVICE_KEY must be set` | Файл `.env` лежит в `dogovor/backend`. В нём есть строки `SUPABASE_URL=...` и `SUPABASE_SERVICE_KEY=...` без лишних пробелов и кавычек вокруг значения. |
| Договор создаётся, но «PDF не сгенерирован» / в Storage нет файла | Создан ли bucket **contracts** в Supabase → Storage? Имя в `.env` (`STORAGE_BUCKET`) совпадает с именем bucket? Используется ли именно **service_role** ключ (не anon)? |
| Ошибка CORS при запросе с фронта к бэкенду | В коде бэкенда уже настроен CORS для `http://localhost:3001`. Если фронт открыт с другого origin (например другой порт) — добавьте его в `allow_origins` в `app/main.py`. |
| `pip: command not found` | Используйте `python3 -m pip install -r requirements.txt` (macOS/Linux) или `py -m pip install -r requirements.txt` (Windows). |
| Порт 8000 занят (`Address already in use`) | Бэкенд уже запущен — можно просто открыть http://localhost:3001. Либо освободить порт: `lsof -ti:8000 | xargs kill -9`, затем снова `./run.sh`. |
| Другой порт для бэкенда | Запуск: `uvicorn app.main:app --reload --port 8001`. Во фронте в `dogovor/frontend/.env.local` укажите `NEXT_PUBLIC_DOGOVOR_API_URL=http://localhost:8001`. |

## Как открыть сайт

Сайт — это фронт на порту **3001**. Чтобы он работал, должны быть запущены **оба** процесса.

1. **На этом компьютере (Mac)**  
   В браузере откройте: **http://localhost:3001**

2. **С планшета/телефона в той же Wi‑Fi сети**  
   В терминале при запуске `npm run dev` выводится строка вида `Network: http://192.168.x.x:3001`. Откройте в браузере планшета именно этот адрес (например `http://192.168.1.94:3001`). Не используйте `localhost` с планшета — он укажет на само устройство.

3. **Если страница не открывается**  
   - «Не удаётся подключиться» / «Сайт недоступен» — не запущен фронт. В терминале: `cd ~/Desktop/cursor/dogovor/frontend && npm run dev`, дождитесь «Ready», затем снова откройте адрес.  
   - Страница открылась, но пишет «Сервер недоступен» или «Не удалось загрузить список кабинетов» — не запущен бэкенд. Запустите его в **другом** терминале (см. шпаргалку ниже), дождитесь «Uvicorn running on…», затем обновите страницу (F5).
