# Договор (Мед. договор) — сканирование паспорта и подписание

Система для кабинетов проверки зрения (ПАЙ ОПТИКС): сканирование паспорта по камере, подстановка данных в шаблон договора, подпись клиента на планшете, сохранение PDF и опциональная отправка в CRM.

---

## 1. Что было в переписке (кратко)

- **Цель:** в кабинете оптометрист выбирает точку → сканирует паспорт клиента → проверяет данные → клиент подписывает → договор в PDF сохраняется, номер уникальный.
- **Три кабинета** с разными адресами; реквизиты ООО общие; **город** подставляется по адресу точки (например Рубинштейна, 10 → Санкт-Петербург).
- **Номер договора:** формат `[Префикс_точки]-[ГГГГ]-[Порядковый]` (например `R10-2025-001`), счётчик — атомарный (Sequence в БД).
- **UI:** «дружелюбный» интерфейс для планшета: крупные шрифты (≥18px), мягкий синий, степпер из 4 шагов, минимум кликов.
- **Безопасность:** не хранить фото паспорта дольше ~30 сек; подпись привязать к идентификатору устройства (UUID планшета).
- Изначально обсуждались: бэкенд на FastAPI, фронт на React (Vite + Tailwind), OCR (Tesseract.js / Regula), S3 для PDF, Supabase для БД. Потом от облачного варианта (Gemini/Cloud) отказались и переносим в этот репозиторий.

---

## 2. Предлагаемый вариант (обоснование)

### 2.1 Общая архитектура

| Компонент | Выбор | Обоснование |
|-----------|--------|-------------|
| **Backend** | **FastAPI (Python)** | Генерация PDF из HTML (Jinja2 + WeasyPrint или wkhtmltopdf) естественна в Python; валидация Pydantic; асинхронность; один сервис для контрактов и нумерации. |
| **Frontend** | **Next.js (React) + Tailwind** | Единый стек с `web/` в этом репо; удобно деплоить на Vercel; при необходимости API Routes как BFF перед FastAPI. Альтернатива — Vite+React, если нужен только SPA без SSR. |
| **БД и хранилище** | **Supabase (PostgreSQL + Storage)** | Уже используется в проекте; один набор ключей; Storage вместо отдельного S3 — меньше интеграций, те же RLS/политики. |
| **Хранение PDF** | **Supabase Storage** | Вместо S3: bucket `contracts`, путь вида `{location_id}/{year}/{contract_number}.pdf`. Достаточно для трёх кабинетов и масштабирования. |
| **Шаблон договора** | **Один Jinja2 HTML-шаблон** | Все точки используют один шаблон; адрес, город, оптометрист, данные пациента — переменные. Добавление нового кабинета = запись в таблице `locations`. |

Итого: в репозитории создаётся папка `dogovor/` с бэкендом (FastAPI) и фронтендом (Next.js), оба работают с одним проектом Supabase (можно тот же, что для `web/`, с отдельными таблицами).

---

### 2.2 Почему не «всё в Next.js»

- **PDF из HTML в Node** возможен (puppeteer, playwright), но тяжёлый и дорогой на serverless (холодный старт, память). Генерация в Python (WeasyPrint/wkhtmltopdf) проще и предсказуемее для «договор по шаблону».
- **Нумерация договоров** с атомарным счётчиком проще и надёжнее в БД (PostgreSQL Sequence) и в одном backend-сервисе, чем размазывать по serverless-функциям.

Поэтому backend договоров — отдельное приложение (FastAPI), которое можно развернуть на Railway, Render, или в одном VPS рядом с фронтом.

---

### 2.3 Почему Supabase Storage, а не S3

- Уже есть Supabase в проекте; не заводить отдельные ключи и bucket’ы в AWS.
- RLS и политики доступа к файлам можно привязать к `auth.uid()` и ролям (оптометрист/админ).
- Для объёма «три кабинета + рост» производительности Storage достаточно.

При необходимости позже можно добавить экспорт в S3 через Supabase или бэкенд.

---

### 2.4 Маппинг «адрес → город»

- В БД таблица **locations**: `id`, `name`, `address`, `city`, `contract_prefix` (например `R10`), привязка к оптометристу (или отдельная таблица `staff`).
- При выборе кабинета фронт получает из API локации; при генерации договора бэкенд подставляет `cabinet_address` и `city` из выбранной локации. Реквизиты ООО — константы в шаблоне или в конфиге бэкенда.

---

### 2.5 Счётчик номеров договоров

- **Вариант A:** один Sequence в PostgreSQL + префикс из `locations.contract_prefix` → номер `R10-2025-00001`. Риск: один общий счётчик для всех точек (если нужны раздельные счётчики по точкам — см. вариант B).
- **Вариант B:** отдельный счётчик на каждую точку (таблица `contract_counters`: `location_id`, `year`, `last_number` с блокировкой при инкременте или Sequence per location). Формат номера тот же.

Для трёх кабинетов достаточно одного Sequence с префиксом; при росте сети можно перейти на счётчик по точке.

## 4. Переменные шаблона договора (из переписки)

| Переменная | Источник |
|------------|----------|
| `contract_number` | БД (Sequence + префикс точки) |
| `current_date` | Сервер (дата визита) |
| `city` | По адресу точки (locations.city) |
| `cabinet_address` | locations.address |
| Реквизиты ООО (ИНН, КПП и т.д.) | Константы в шаблоне/конфиге |
| `staff_fio` | Выбранный оптометрист (из списка по городу кабинета) |
| `staff_signature_url` | URL подписи оптометриста для автоподстановки в договор (загрузить позже) |
| `patient_fio`, `patient_birth_date`, `passport_series`, `passport_number`, `passport_issued_by`, `passport_date`, `reg_address` | OCR паспорта + форма проверки |

Текст договора из вашего .docx нужно вставить в `templates/contract.html`, заменив статичные значения на `{{ variable_name }}`.

## 5. Безопасность (как в переписке)

- **Фото паспорта:** не сохраняем; после извлечения текста (OCR) изображение удаляется с фронта в течение ~30 сек (таймер + очистка blob/object URL).
- **Подпись:** сохраняем в PDF; при необходимости привязывать к `device_uuid` (с фронта) и сохранять в метаданных договора в БД.
- **Доступ к API:** только с авторизованных фронтов (CORS); в перспективе — Supabase Auth (оптометрист по логину получает свою точку).

## 6. Как запустить (текущее состояние)

1. **Supabase:** создать проект (или использовать существующий). В SQL Editor выполнить весь файл `dogovor/supabase/001_contracts_schema.sql` (таблицы + функция + сиды). В Storage создать bucket `contracts` (при необходимости сделать его публичным для доступа по ссылке).
2. **Бэкенд:** `cd dogovor/backend`, скопировать `.env.example` в `.env`, подставить `SUPABASE_URL` и `SUPABASE_SERVICE_KEY`. Затем:
   ```bash
   pip install -r requirements.txt
   uvicorn app.main:app --reload --port 8000
   ```
3. **Фронт:** `cd dogovor/frontend`, установить зависимости и запустить:
   ```bash
   npm install
   npm run dev
   ```
   Откроется http://localhost:3001 — шаг «Кабинет и врач» должен подтянуть список из API. Бэкенд должен быть запущен на порту 8000.

## 7. Следующие шаги (доработки)

1. **HTML-шаблон и PDF:** вставить текст из ПАЙ_ОПТИКС_МЕД_ДОГ.docx в `backend/templates/contract.html`, расставить переменные Jinja2; подключить генерацию PDF (WeasyPrint/wkhtmltopdf) и загрузку в Supabase Storage.
2. **Степпер (шаги 2–4):** камера/OCR, проверка данных в карточках, canvas подписи и вызов `POST /contracts/generate` с данными пациента и подписью.
3. **OCR:** на MVP — Tesseract.js в браузере; при необходимости Regula или серверный OCR.
4. **Подписи оптометристов:** при вставке подписи врача в PDF — напомнить заказчику прислать отсканированные подписи; загрузить в Storage и проставить `signature_image_url` в `dogovor_staff`. См. `docs/LOCATIONS_AND_STAFF.md`.

После этого можно подключать webhook «договор подписан» для будущей CRM (POST от бэкенда при успешном сохранении PDF).

---

## 8. Итог

- **Backend:** FastAPI (Python), один шаблон Jinja2, PDF в Supabase Storage, нумерация через БД.
- **Frontend:** Next.js + Tailwind, степпер, OCR на клиенте (Tesseract.js для старта), canvas для подписи.
- **БД/Storage:** Supabase (таблицы для локаций и договоров, Storage для PDF).
- **Три кабинета:** одна конфигурация в таблице `locations`; город и префикс номера задаются там же.

Такой вариант даёт единый стек с текущим репозиторием, минимум новых сервисов и возможность масштабировать число кабинетов без дублирования шаблонов.
