# Play Pye — Архитектура системы

Краткий обзор компонентов и потоков данных.

**Порядок разработки:** сначала игра (данные из статики, прогресс в localStorage), затем подключение Supabase, Notion и Miro без смены контракта данных.

---

## Слои приложения

```
┌─────────────────────────────────────────────────────────────────┐
│  Client (Next.js SPA)                                            │
│  ├── UI Shell (React + Framer Motion): меню, карта мира, HUD     │
│  └── Game Canvas (Phaser.js): физика, уровни, персонаж           │
└─────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────┐
│  Supabase                                                        │
│  ├── Auth (гость = session_token, пользователь = auth.uid)        │
│  ├── DB: profiles, game_sessions, player_progress, levels, quests│
│  └── Storage: ассеты уровней (опционально)                       │
└─────────────────────────────────────────────────────────────────┘
                                    ▲
                                    │ sync + validate
┌─────────────────────────────────────────────────────────────────┐
│  Sync Service (Edge Functions / Cron)                            │
│  ├── Notion API → quests (парсинг, валидация)                    │
│  ├── Miro API → Mapper → level_data → levels (валидация)         │

                                    ▲
                    ┌───────────────┴───────────────┐
                    │                               │
              ┌─────┴─────┐                   ┌─────┴─────┐
              │  Notion   │                   │   Miro    │

              └───────────┘                   └───────────┘
```

## Роли данных

| Компонент | Роль |
|-----------|------|
| **Notion** | Источник правды для квестов (заголовки, описания, цели, награды). |
| **Miro** | Источник правды для визуала уровней (рамки, формы → платформы, spawn, exit). |
| **Supabase** | Кеш провалидированных уровней и квестов; профили; прогресс; сессии. |
| **Data Validator** | Проверка контента до записи в БД; ошибки не попадают в клиент. |
| **Mapper (Miro → Level)** | Преобразование элементов доски в канонический `level_data` (JSON). |

---

## Игровой цикл (упрощённо)

1. Пользователь заходит на сайт → создаётся/читается `session_token` (гость) или выполняется вход (Supabase Auth).
2. Карта мира загружает список уровней из `levels` (только `validation_status = 'valid'`, `published_at` не null).
3. Выбор уровня → загрузка `level_data` → сцена Phaser строит уровень, spawn игрока.
4. Прохождение уровня → сохранение в `player_progress` по `session_id` (звёзды, время).
5. При желании «сохранить в облако» — регистрация/вход → привязка `game_sessions.user_id`, merge прогресса.

## Ограничения и оптимизация (Free Tier)

- Notion/Miro: кешировать ответы в Supabase, не опрашивать чаще необходимого (cron раз в час или по вебхуку).
- Supabase: RLS так настроен, что клиент читает только нужные таблицы; запись в `levels`/`quests`/`sync_log` — только через service role.
- Клиент: запрашивать только опубликованные уровни и валидные квесты; при ошибке/пустом ответе — заглушка, без падения.
