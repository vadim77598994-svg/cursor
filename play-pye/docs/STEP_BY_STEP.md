# Play Pye — Пошаговый план реализации

**Стратегия:** сначала делаем саму игру (движок, уровни из статики, прогресс в абстракции), затем подключаем Notion, Miro и Supabase. Канонический формат данных (`level_data`, квесты) задаём с первого шага и не меняем при интеграции.

---

## Контракт данных (не менять при подключении CMS)

Игра с самого начала потребляет только эти форматы. Моки и статика — в том же виде.

**Уровень (`level_data`):**
```ts
// lib/game/types.ts или аналогично
interface LevelData {
  version: number;
  spawn: { x: number; y: number };
  exit: { x: number; y: number };
  platforms: Array<{ id: string; x: number; y: number; width: number; height: number; type: string }>;
  triggers?: Array<{ id: string; x: number; y: number; width: number; height: number; kind: string; payload?: object }>;
  assets?: { tileset_url?: string; background_url?: string };
}
```

**Список уровней (карта мира):**
```ts
interface LevelMeta {
  slug: string;
  name: string;
  description?: string;
  sort_order: number;
}
```

**Квест (для последующей интеграции с Notion):**
```ts
interface QuestData {
  slug: string;
  title: string;
  description?: string;
  objectives: string[];
  rewards?: string[];
  level_slug?: string;
}
```

**Прогресс (абстракция «сохранить/загрузить»):**
```ts
interface PlayerProgressEntry {
  level_slug: string;
  stars: number;
  completed_at?: string; // ISO
  best_time_ms?: number;
  payload?: Record<string, unknown>;
}
// Сервис: getProgress(sessionId): Promise<PlayerProgressEntry[]>
//         saveProgress(sessionId, entry): Promise<void>
// Сначала реализация — localStorage или in-memory; потом — Supabase.
```

# ЧАСТЬ A: Игра сначала

Цель: играбельный платформер с картой уровней и сохранением прогресса без внешних API. Все данные — из статики (JSON/константы) в каноническом формате.

---

## A.1 Каркас проекта

| # | Шаг | Задачи | Критерий готовности |
|---|-----|--------|----------------------|
| A.1.1 | Next.js + структура | Инициализация Next.js (App Router). Папки: `app`, `components`, `lib`, `game` (Phaser), `data` (моки уровней/квестов), `styles`. | `npm run build` проходит, главная страница открывается. |
| A.1.2 | Контракт данных в коде | Файл `lib/game/types.ts` (или аналог): типы `LevelData`, `LevelMeta`, `QuestData`, `PlayerProgressEntry`. Файл `data/levels/` с 1–2 уровнями в JSON в формате `level_data`. | Типы используются в коде; уровень загружается из JSON. |
| A.1.3 | Абстракция прогресса | Сервис `lib/game/progress.ts`: `getProgress(sessionId)`, `saveProgress(sessionId, entry)`. Реализация — localStorage + ключ по `sessionId` (пока простой UUID в localStorage). | Игра может сохранять/загружать прогресс по уровню без БД. |

## A.2 Игровой движок (Phaser)

| # | Шаг | Задачи | Критерий готовности |
|---|-----|--------|----------------------|
| A.2.1 | Phaser в Next.js | Подключить Phaser 3. Сцена Boot + сцена Level. Рендер в canvas (useRef, useEffect), размер под layout. | На странице отображается пустая сцена Phaser без ошибок. |
| A.2.2 | Загрузка уровня из данных | Сцена Level принимает `LevelData`. Отрисовка платформ из `platforms`, spawn игрока, зона выхода (exit). Данные брать из `data/levels/*.json` или из импорта. | Игрок появляется в spawn, видит платформы и exit. |
| A.2.3 | Физика и управление | Персонаж: движение влево/вправо, прыжок, коллизия с платформами. Камера следует за игроком. | Марио-стайл движение и прыжки работают. |
| A.2.4 | Завершение уровня и сохранение | При достижении exit — вызов сервиса прогресса: `saveProgress(sessionId, { level_slug, stars, completed_at, best_time_ms })`. Экран «Уровень пройден» (звёзды/время). | Прохождение уровня сохраняется и отображается при повторном заходе. |

## Итог части A

- Игра полностью играбельна: карта → уровень → прогресс в localStorage.
- Все данные в каноническом формате; источник — только статика (JSON/константы).
- Прогресс идёт через абстракцию `getProgress` / `saveProgress`, готовую к замене на Supabase.

# ЧАСТЬ B: Подключаем всё остальное

Цель: заменить статику на Supabase/Notion/Miro; добавить Auth, гостевой режим, синк и валидацию.

---

## B.1 Supabase: БД и Auth

| # | Шаг | Задачи | Критерий готовности |
|---|-----|--------|----------------------|
| B.1.1 | Схема БД | Применить `supabase/schema.sql`: `profiles`, `game_sessions`, `player_progress`, `levels`, `quests`, `sync_log`. RLS по документации. | Миграции применены, RLS включён. |
| B.1.2 | Supabase Auth + профиль | Подключить Supabase Auth (email или OAuth). Таблица `profiles`, триггер на создание профиля при регистрации. Экран входа/регистрации. | Пользователь может зарегистрироваться и видеть профиль. |
| B.1.3 | Гостевой режим и сессии | Генерация `session_token` (UUID) в localStorage при первом заходе. Создание/обновление `game_sessions` (user_id = null). API или Edge Function для создания сессии без auth. | Гость получает session_token, в БД есть запись в `game_sessions`. |

## B.2 Уровни и прогресс из Supabase

| # | Шаг | Задачи | Критерий готовности |
|---|-----|--------|----------------------|
| B.2.1 | Загрузка уровней из БД | API или Supabase client: запрос списка уровней (`levels` с `validation_status = 'valid'`, `published_at` не null). Формат ответа — тот же `LevelMeta` + `level_data`. Заменить в карте мира и в сцене Level чтение из статики на запрос к API. | Карта мира и уровень берут данные из Supabase. |
| B.2.2 | Реализация прогресса в Supabase | Реализация `getProgress`/`saveProgress`: через `player_progress` и `game_sessions` (session_id из БД по session_token). Сохранять при завершении уровня; при первом заходе гостя — создавать сессию, затем прогресс. | Прогресс гостя сохраняется в БД и отображается на карте. |
| B.2.3 | Привязка прогресса к аккаунту | При входе после гостевой игры: по `session_token` найти сессию, предложить «Сохранить прогресс в облако». После логина — обновить `game_sessions.user_id`, merge прогресса по level_slug (лучший результат). | Гость может залогиниться и перенести прогресс в профиль. |

## B.3 Синхронизация и валидация (Notion + Miro)

| # | Шаг | Задачи | Критерий готовности |
|---|-----|--------|----------------------|
| B.3.1 | Sync Notion → quests | Edge Function или API route: запрос к Notion (DB или страницы квестов), парсинг в формат `QuestData`. Валидатор: непустые title/description, обязательные поля. Upsert в `quests` с `validation_status` и `validation_errors`. | В `quests` появляются записи с `validation_status = 'valid'`. |
| B.3.2 | Miro Mapper + валидация уровней | Правила маппинга: элементы Miro (frames/shapes) → канонический `level_data`. Валидатор: spawn/exit, размеры платформ, проходимость (опционально). Запись в `levels` с `validation_errors` при ошибках. | Синк из Miro заполняет `levels.level_data`; невалидные уровни не публикуются. |
| B.3.3 | Квесты в игре из БД | Загрузка квестов из `quests` по `level_slug`. В игре — те же типы `QuestData`; источник — API вместо статики. Триггеры в уровне обновляют прогресс квеста (сохранение в `player_progress.payload` или отдельная таблица — по выбору). | В уровне отображаются квесты из Notion. |
| B.3.4 | Валидатор и лимиты | Расширить валидатор: битые ссылки, пустые блоки Notion; для уровней — битые asset URL. Логирование в `sync_log`. Кеширование и частота синка в рамках Free Tier. | Невалидный контент не публикуется; синк укладывается в лимиты. |
